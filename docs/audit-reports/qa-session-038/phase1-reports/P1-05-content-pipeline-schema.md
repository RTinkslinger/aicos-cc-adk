## 9. Critical Findings

### Finding #1: Two Versions of Schema In Production

**Severity:** 🟡 Medium

The digests corpus contains **two incompatible schema versions**:

- **v3 (Legacy):** Files: ben-horowitz-*, from-writing-code-*, how-to-code-*, the-powerful-*, the-saas-*
  - Uses: `core_argument`, `evidence`, `framework` (singular)
  - Uses: `key_question_impact` instead of `key_question`
  - Has: `summary`, `key_insights`, `topic_map` (extra fields)

- **v4 (Current):** Files: agi-po-shen-*, ben-marc-*, cursor-*, design-*, india-*, nuclear-*, startup-*
  - Uses: `core_arguments`, `data_points`, `frameworks` (plural)
  - Uses: `key_question` (correct)
  - Missing: `summary`, `key_insights`, `topic_map`

**Root Cause:** Different LLM prompts/templates across sessions. Session 027 attempted fix (singular → plural conversion) but SKILL.md template wasn't updated consistently.

**Current Impact:** ✅ MITIGATED — normalization in digests.ts handles both versions correctly.

**Future Risk:** If normalizations are removed or changed without updating BOTH skill template AND digests.ts, rendering will break.

### Finding #2: Normalization is Defense-in-Depth, Not Schema Enforcement

**Severity:** 🟡 Medium

The architecture assumes JSON can be malformed and uses runtime normalization to fix it. This is good defensive coding, but it masks underlying schema drift.

**Better Approach:** 
1. Update SKILL.md to produce only v4 schema
2. Regenerate v3 files through skill or remove them
3. Remove normalizations from digests.ts (keep only enum mappings)
4. Add strict type checking to detect drift earlier

**Current State:** Works but requires constant vigilance.

### Finding #3: publish_digest.py Doesn't Validate Schema

**Severity:** 🟢 Low (mitigated by digests.ts normalization)

publish_digest.py writes any JSON without schema validation. If LLM produces invalid JSON (e.g., missing `essence_notes`), the file still saves.

**Risk:** DigestClient might break on malformed JSON.

**Current Mitigation:** 🟢 Sufficient — digests.ts normalizes missing nested objects before rendering.

**Recommendation:** Add optional JSON schema validation in publish_digest.py (e.g., via jsonschema library).

### Finding #4: Missing Fields Never Added to types.ts

**Severity:** 🟡 Medium

v3 files contain `summary`, `key_insights`, `topic_map` that are never rendered because they're not in DigestClient and not in the types.ts interface.

**Implication:** These fields are produced by LLM but wasted.

**Options:**
1. Remove from SKILL.md (simplify template)
2. Add to types.ts and render (add to DigestClient)
3. Map to different names (e.g., `summary` → description in metadata)

**Recommendation:** Remove from SKILL.md — currently not used by renderer.
