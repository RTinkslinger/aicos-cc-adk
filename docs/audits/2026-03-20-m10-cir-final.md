### Issues Found

1. **Queue backlog (P1):** Processor reads 10 items/min. Burst activity from entity operations creates backlogs. Current backlog: 3,483 messages (~5.8 hours to drain). **Fix:** Increase batch size to 50-100, or increase frequency to every 10-15 seconds.

2. **No dead-letter queue (P2):** If `process_cir_queue_item` fails, the message is already archived (read+archive in same CTE). Failed items are logged but cannot be retried. **Fix:** Separate read from archive — only archive after successful processing.

3. **Duplicate QUEUE_PROCESSED entries (P3):** The queue processor's CTE reads messages with visibility timeout 60s, but if processing takes >60s, the same message becomes visible again. Evidence: source_id=6 has 5+ QUEUE_PROCESSED entries. **Fix:** Use longer visibility timeout (300s) or add dedup check.

4. **Decay not yet validated in production (P3):** All connections are <1 day old. The decay cron runs daily at 21:30 UTC but has zero run history. Will be validated organically within 24 hours.

### Recommendations

- **Immediate:** Increase queue processor batch size from 10 to 100 and visibility timeout from 60s to 300s
- **Short-term:** Add dead-letter queue pattern (separate read from archive in processor CTE)
- **Medium-term:** Add connection refresh mechanism — when new evidence arrives for a decayed connection, boost strength back up (currently only decay, no reinforcement)
- **Monitor:** Check queue depth daily for first week. If backlog persists after L3 burst drains, the processor needs permanent scaling.
