## L25-28: Fix Score Compression

### Approach Tested
1. **Sigmoid stretch** (k=1.5, m=8.8): 36.5% in bucket 8 -- too many scores piled at the asymptote
2. **Linear min-max** (5.65-9.65 to 1-10): 38.3% in bucket 10 -- raw data clusters at top
3. **Percentile normalization** (percent_rank): Perfect 9.8% per bucket -- selected

### Solution Implemented
Created `normalize_all_scores()` function that:
1. Recomputes raw scores via `compute_user_priority_score()`
2. Applies `percent_rank()` window function over all actions
3. Maps percentile rank to 1-10 scale: `1 + 9 * percent_rank`

This preserves the relative ordering from the 7-factor model while guaranteeing a uniform distribution.

## L29-32: Outcome Feedback Loop

### Created: `update_preference_from_rating(action_id, rating)`
- **Input:** action_id (int), rating (1-5 scale: 1=terrible, 2=bad, 3=neutral, 4=good, 5=great)
- **Rating to adjustment mapping:** `(rating - 3) * 0.2` = -0.4 to +0.4
- **Updates 4 dimensions:** action_type, priority, source, thesis_connection
- **Learning:** Exponential moving average with learning_rate=0.3
  - `new_weight = 0.7 * old_weight + 0.3 * adjustment`
- **Records outcome** in `action_outcomes` table
- **Returns:** JSONB with all updated dimensions and their before/after adjustments

### Testing
- Rating action 1 as "Good" (4/5): correctly applied +0.2 to Pipeline Action, P0, Thesis Research
- Rating action 2 as "Bad" (2/5): correctly applied -0.2 to Portfolio Check-in, P0, Thesis Research
- EMA updates work correctly (existing weights blended, new entries created)
- Test outcomes cleaned up

### Wiring
The existing `compute_user_priority_score()` already reads from `preference_weight_adjustments` table. New ratings flow through:
```
User rates action -> update_preference_from_rating()
  -> preference_weight_adjustments (EMA update)
  -> action_outcomes (historical record)
  -> next normalize_all_scores() run picks up new weights
```
