### Intelligence Quality Findings

1. **Staleness boost is inverted.** Actions get HIGHER scores the OLDER they are (log(days_proposed+1) up to +0.8). The design intent was to surface forgotten items, but the effect is that stale P0 items from 15 days ago crowd out fresh P1 items from today. The staleness boost should decay, not grow. This is the single biggest scoring bug.

2. **Research actions score too high.** f_time=1.0, f_stakeholder=1.0, f_effort=1.0, f_novelty=1.0 for research tasks is unrealistic. The scoring function gives maximum factor scores to research because the default/fallback values are 1.0 when no specific data exists.

3. **User-assigned priority is ignored.** Action 103 is marked P2 by the user but scores 9.57. The scoring model has no factor for user-assigned priority (P0/P1/P2/P3). This contradicts the entire purpose of the priority field.

4. **Score compression has shifted, not improved.** Previously 60% in bucket 8. Now 55.5% below 6.0 and only 6.3% above 9.5. The distribution went from compressed-high to bimodal, which is arguably worse — most actions are either very high (top 20) or very low (bottom 71).

5. **Thesis connections are accurate.** Top thesis (Agentic AI Infrastructure) has 33 connected actions and 82 target entity connections. SaaS Death has 32 actions. The ILIKE matching works correctly. Healthcare AI and USTOL have only 2 actions each — low coverage but not incorrect.

6. **M7 strategic_score still not read by M5.** Confirmed: `compute_user_priority_score()` does not reference `strategic_score`. The strategic scores exist (9.70 on Cultured Computers, 9.17 on Unifize) but are completely ignored in ranking. The feedback loop is still broken.

## L77-80: CROSS-MACHINE CONFLICT AUDIT

### Trigger Conflicts

**CRITICAL: interactions table has 12 triggers (6 INSERT, 6 UPDATE)**

| Trigger | Event | Action |
|---------|-------|--------|
| cir_interactions | INSERT | cir_change_detected() |
| embed_interactions_on_insert | INSERT | util.queue_embeddings('embedding_input', 'embedding') |
| interactions_embedding_queue | INSERT | util.queue_embeddings('embedding_input_interactions', 'embedding') |
| trg_interactions_embedding_input | INSERT (BEFORE) | populate_interactions_embedding_input() |
| trg_interactions_fts | INSERT (BEFORE) | interactions_fts_trigger() |
| + 7 more on UPDATE | UPDATE | Various |

**Problem:** `embed_interactions_on_insert` and `interactions_embedding_queue` are DUPLICATE triggers — both queue embeddings on INSERT, but with DIFFERENT embedding_input column names (`'embedding_input'` vs `'embedding_input_interactions'`). One references a column that may not exist, causing silent failures or double-queuing.

### Function Signature Mismatch

**CIR -> M5 broken link:** `process_cir_queue_item` calls `compute_user_priority_score(v_action_row)` passing a ROW type, but the CIR rule generates events with `target_action = 'process_new_action'` that expects an integer action ID. The function was FIXED in the code (fetches full row first) but 7 errors logged on 2026-03-21 suggest the fix was deployed AFTER the actions (IDs 118-124) were inserted. Those 7 actions were NEVER scored by CIR propagation.

### pg_cron Job Conflicts

| Job | Schedule | Concern |
|-----|----------|---------|
| process-embeddings | 10 seconds | Running every 10s — 8,640 executions/day. Healthy but aggressive. |
| cir-queue-processor | every minute | 1,440 executions/day. Each reads up to 200 messages. Fine. |
| cir-staleness-check | hourly | Inserts into cir_propagation_log for every stale action. With 77 stale checks and growing, this is a log pollution source. |
| normalize-scores | every 6 hours | Calls normalize_all_scores(). Healthy. |

### Log Pollution

**cir_propagation_log: 46,267 rows in ~24 hours.** Breaking down:
- 28,806 UPDATE events (62%)

- 6,362 INDICATOR_UPDATE (14%)

**No retention policy.** At this rate: ~1.4M rows/month, ~17M rows/year. The table will become a performance drag. Needs a 7-day retention cron job.

## L81-84: PERFORMANCE DEEP DIVE

### Query Performance (EXPLAIN ANALYZE results)

| Query/View | Time (ms) | Verdict |
|------------|-----------|---------|
| `user_triage_queue LIMIT 20` | **946.3** | CRITICAL. Recalculates `compute_user_priority_score()` per row. This is the WebFront /actions page query. |
| `hybrid_search('AI infrastructure')` | 19.7 | Acceptable. FTS-only (no embedding). |
| `entity_relationships LIMIT 20` | 11.6 | Acceptable but with cross-join potential. |

| `network ORDER BY priority LIMIT 50` | 3.2 | Good. Seq scan on 3,525 rows but fast enough. |
| `action_score_breakdown LIMIT 20` | 2.2 | Good. Uses stored user_priority_score. |
| `megamind_dashboard LIMIT 20` | 0.8 | Excellent. Simple join. |
| `obligation_dashboard` | 0.3 | Excellent. Tiny table. |

### Top 5 Performance Issues

1. **user_triage_queue: 946ms** — The view sorts by `compute_user_priority_score(aq.*)` DESC, which calls the function for EVERY row. The function does 6 subqueries (preference_weight_adjustments, portfolio name match, etc.) per action. With 103 proposed actions, that is 618+ subqueries per page load. **FIX:** Use the stored `user_priority_score` column instead of recalculating, or use the `action_scores` materialized view.

2. **cir_propagation_log: 15MB and growing** — 46K rows in 24h, no retention. Seq scan for stale checks. **FIX:** Add retention cron (DELETE WHERE created_at < NOW() - interval '7 days'), add index on (event_type, created_at).

3. **network seq_scan: 28,963** — The network table (63MB) has 28,963 seq scans, which is high. Missing index on `e_e_priority` for the common ORDER BY query. **FIX:** `CREATE INDEX idx_network_priority ON network(e_e_priority DESC NULLS LAST)`.

4. **thesis_threads seq_scan: 54,078** — 54K seq scans on a table with only 8 rows. Every function that touches thesis threads does a seq scan. Not a performance issue at 8 rows, but indicates that views/functions reference thesis_threads excessively — the scoring function likely joins thesis_threads on every action evaluation.

5. **actions_queue seq_scan: 20,623** — High seq scan count on 128 rows. The `idx_actions_status` index has 5,361 scans vs 20,623 seq scans — many queries don't filter by status. **FIX:** Ensure all WebFront queries use status filter.

### Unused Indexes

- `entity_connections_pkey`: 14,382 scans but only 45 tuples read — the unique constraint index fires on every upsert check but finds nothing (expected for ON CONFLICT).
- `a_cir_queue`: 14,465 rows, 0 idx_scans — archive table with no queries.

### Performance Score: 4.5/10
