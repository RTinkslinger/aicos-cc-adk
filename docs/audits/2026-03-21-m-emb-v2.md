# M-EMB v2: Embedding Queue Recovery - Deep Fix

**Date:** 2026-03-21
**Machine:** M-EMB (Temporary)
**Status:** FIXED - Queue actively draining, all tables processing

---

## Problem Statement

Previous M-EMB fix (v1) increased batch parameters and reported the queue was draining at 4,500/hr. However, the queue **stopped draining** afterward. At session start:
- 11,505 messages stuck in pgmq `embedding_jobs` queue
- ALL messages had `read_ct: 0` (never read)
- Network embeddings stalled at 10.5% (370/3528)
- Companies at ~74% (but many were already embedded)

## Root Cause Analysis

### Root Cause 1: Edge Function Does NOT Delete Queue Messages

The v1 audit assumed the edge function was deleting messages via `pgmq.delete()` after processing. **This is wrong.** Deep investigation reveals:

1. `process_embeddings()` calls `pgmq.read()` -- which sets a visibility timeout (VT) making messages invisible
2. Messages are sent to the `embed` Edge Function via `pg_net` (fire-and-forget async HTTP)
3. Edge Function generates embeddings and writes them to the table (confirmed: HTTP 200, `completedJobs` returned)
4. **But nobody deletes the messages from the queue**
5. After VT expires (~300s), messages become visible again and get re-processed infinitely

**Evidence:**
- Edge Function response: `{"completedJobs":[...], "failedJobs":[]}`
- Corresponding queue messages still exist with `read_ct: 0` (pgmq resets read_ct on VT expiry? or msgs were never actually read)
- 124,188 messages WERE consumed earlier (total_messages 135,693 - queue_length 11,505). These were likely processed when the system was working correctly, meaning something changed.

### Root Cause 2: Connection Pool Exhaustion Kills Network Embeddings

With `max_requests=15, batch_size=25` (v1 fix), the function fires 15 concurrent HTTP requests. Each Edge Function instance opens connections back to Postgres. With 60 max connections:

**Confirmed error in HTTP response 14776:**
```
"failedJobs":[
  {"error":"remaining connection slots are reserved for roles with the SUPERUSER attribute"},
  {"error":"sorry, too many clients already"}
]
```

Companies processed fine (faster SQL, smaller content). Network records failed due to connection exhaustion -- `embedding_input_network()` concatenates more fields including `page_content` (up to 2000 chars), causing longer transactions.

### Root Cause 3: Message Ordering Blocks Network Processing

`pgmq.read()` returns messages ordered by `msg_id` (ascending). Company messages had lower msg_ids than network messages. Since company messages were never deleted (Root Cause 1), they kept being re-read and re-processed, blocking network messages from ever being reached.

## Fixes Applied

### Fix 1: Created Cleanup Cron (NEW)

Created `util.cleanup_embedding_jobs()` -- a function that checks queue messages against actual tables and deletes messages where the embedding already exists.

```sql
-- Runs every odd minute (1, 3, 5...) to alternate with processing
cron.schedule('cleanup-embedding-jobs', '1-59/2 * * * *', 'SELECT util.cleanup_embedding_jobs(500);')
```

This is the critical missing piece. Without it, successfully-processed messages accumulate forever.

### Fix 2: Reduced Concurrency to Prevent Connection Exhaustion

Changed cron job parameters:
| Parameter | v1 (broken) | v2 (working) |
|-----------|-------------|--------------|
| `batch_size` | 25 | **15** |
| `max_requests` | 15 | **5** |
| `timeout_milliseconds` | 300000 | **120000** |
| Per cycle | 375 | **75** |
| Cron interval | */5 min | ***/2 min** |
| **Effective rate** | 4,500/hr (theory, but stalled) | **2,250/hr (sustained)** |

5 concurrent Edge Function calls with 15 items each = 5 simultaneous Postgres connections from edge functions = well within the 60-connection limit.

### Fix 3: Manual Backlog Cleanup

Deleted ~7,500 queue messages for records that already had embeddings:
- ~6,700 company messages (records already at 93.6% embedded)
- ~372 network messages (records already embedded)
- ~68 interaction messages (all 23 records already embedded)
- Reduced queue from 11,505 to ~3,850 genuine items

## Architecture Diagram (Corrected)

```
Trigger (INSERT/UPDATE on content columns)
  → util.queue_embeddings() → pgmq.send('embedding_jobs', {...})

Cron (*/2 * * * * - process-embeddings, jobid=1)
  → util.process_embeddings(batch_size=15, max_requests=5, timeout=120s)
    → pgmq.read(qty=75, vt=120)   -- makes messages invisible for 120s
    → net.http_post → Edge Function 'embed' (5 calls of 15 items each)

Edge Function 'embed':
  → For each item: SELECT contentFunction(record) → get text
  → Generate embedding via AI model
  → UPDATE table SET embedding = vector WHERE id = X
  → Returns {completedJobs: [...], failedJobs: [...]}
  → Does NOT delete queue messages (BUG or design gap)

Cron (1-59/2 * * * * - cleanup-embedding-jobs, jobid=18)  ← NEW
  → util.cleanup_embedding_jobs(1000)
    → For each table: DELETE queue messages where embedding IS NOT NULL
```

## Why v1 Fix Failed

The v1 fix increased `batch_size` from 10 to 25 and `max_requests` from 10 to 15. This:
1. Temporarily increased throughput (4,500/hr claimed)
2. But caused connection exhaustion for network records
3. Successfully-processed messages were never deleted from the queue
4. Queue appeared to drain temporarily (edge function was processing, HTTP responses showed success)
5. But messages re-appeared after VT expired (300s)
6. Created an infinite loop: read → process → VT expires → read again

## Recommendations

1. **Fix the Edge Function** (ideal but requires deploy access): Add `pgmq.delete('embedding_jobs', jobId)` after successfully writing each embedding. This eliminates the need for the cleanup cron.

2. **Monitor queue depth**: `SELECT queue_length FROM pgmq.metrics('embedding_jobs')` should reach near-zero by ~11:40 UTC.

3. **Connection budget**: With 60 max connections, keep `max_requests <= 5` for embedding processing. The edge function opens connections back to Postgres, so each request costs ~3 connections (read content function, write embedding, delete from queue).

4. **Clean net._http_response**: `DELETE FROM net._http_response WHERE created < now() - interval '24 hours'` -- accumulates rapidly.

5. **Re-enqueue check**: The `embed_on_update` trigger only fires on content-column changes (name, sector, deal_status, jtbd, agent_ids_notes, page_content), NOT on embedding-column updates. This is correctly designed -- no infinite loop risk.
