## Architecture Assessment

### Data Fetching Pattern
All pages use **React Server Components** for initial data loading with `Promise.all()` for parallel fetching. No N+1 query issues detected in the modified pages. The `fetchAllScoreBreakdowns()` batch query prevents per-card API calls.

### Performance Notes
- All new data fetching is server-side (no client-side fetches for initial load)
- Breakdown data is passed as a serialized lookup object (not re-fetched per card)
- CIR dashboard is a single RPC call returning all data
- Obligation overdue query uses the materialized `obligation_dashboard` view (fast)
- QuickRateModal is client-side only (rendered on demand, not in initial bundle)

### Component Hierarchy
```
Server: actions/page.tsx
  ├── fetchUserTriageQueue(), fetchActionScoresRanked(), fetchAllScoreBreakdowns()
  └── Client: ActionsPageClient
       ├── ActionsBucketList → BucketSection → InlineScoreBreakdown
       ├── ActionRowCompact (with breakdown factor dots)
       └── QuickRateModal (lazy, on demand)

Server: comms/page.tsx
  ├── fetchObligations(), fetchOverdueObligations()
  └── Overdue Alert Banner (enhanced, server-rendered)

Server: strategy/page.tsx
  ├── fetchCirDashboard()
  └── CirHealthPanel (server-rendered)
```

### Bundle Impact
- New client components: InlineScoreBreakdown (~2KB), QuickRateModal (~4KB)
- New server action: outcomes.ts (~1KB)
- New query functions: ~3KB added to queries.ts
- Net: minimal bundle increase, most work is server-side
