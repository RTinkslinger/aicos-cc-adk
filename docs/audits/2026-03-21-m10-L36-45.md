## L36-37: EMERGENCY — Cron Job Connection Pool Fix

### Problem
`process-embeddings` cron ran every 59 seconds (7,709 runs/day), exhausting the Supabase connection pool. This caused 3 cron jobs to FAIL at 0% success rate and 3 more to degrade:

| Job | 24h Success Rate | Status |
|-----|-----------------|--------|
| normalize-scores | 0.0% | FAILING |
| daily_strategic_assessment | 0.0% | FAILING |
| cir-staleness-refresh | 0.0% | FAILING |
| cir-matview-refresh | 86.5% | DEGRADED |
| cir-staleness-check | 83.3% | DEGRADED |
| cir-queue-processor | 81.8% | DEGRADED |

### Fix Applied
1. **Schedule change:** `process-embeddings` from `59 seconds` to `*/5 * * * *` (every 5 min). Reduces runs from ~7,700/day to ~288/day (96% reduction). Same batch throughput — function already processes in batches of 10x10.
2. **Advisory lock:** Added `pg_try_advisory_lock()` to prevent concurrent runs from stacking connections.
3. **Timeouts:** `SET LOCAL statement_timeout = '30s'` and `SET LOCAL lock_timeout = '5s'` to prevent hung connections.
4. **Manual kick:** All 3 FAILING jobs executed manually to verify they work post-fix.

### Verification
- `normalize-scores`: 136 scores updated, range 1.00-10.00
- `daily_strategic_assessment`: executed successfully
- `cir-staleness-refresh`: refreshed 1 thesis entity
- All jobs showing 0 failures in last hour post-fix
