## Loop Details

### Loop 1 (L46-47): Embedding Queue Monitoring + Phantom Cleanup

**Root cause diagnosed:** M12 queued 15,215 embedding jobs. The `process_embeddings` cron calls the `embed` Edge Function which is failing with "connection failed" (540 failures/24h). The queue has 93% cron "success" rate -- but those successes are the advisory lock skip or empty read path. All 15,015 messages have read_ct=0 (never actually attempted). Queue is growing, not draining.

**Second bug found:** `process_cir_queue_batch` builds entity keys as `source_table:source_id` using the raw PostgreSQL table name (e.g., `companies:123`). But all other CIR functions use canonical types (`company:123`). This created 4,650 phantom cir_state entries that polluted entity counts and staleness calculations.

**Actions:**
- **L46:** Created `embedding_queue_health()` function -- returns queue depth, table breakdown, processing rate, estimated drain time, embedding coverage, cron success/failure stats, and health grade (CLEAR/HEALTHY/PROCESSING/BACKLOGGED/FAILING/CRITICAL)
- **L47:** Created `cir_normalize_entity_type()` -- maps raw table names to canonical types. Patched `process_cir_queue_batch` to use it. Cleaned 4,650 phantom entries from cir_state.

**Result:** Full embedding queue observability. Phantom entry creation permanently fixed. cir_state now has only canonical entity types.

### Loop 2 (L48-49): Intelligent Propagation

**Problem:** All CIR queue items were processed equally -- a bulk company metadata update got the same priority as a strategic cascade event from Megamind. Under load, high-value signals could wait behind thousands of low-value bulk updates.

**Actions:**
- **L48:** Created `propagation_priority_score(event)` -- scores each event 0-100 based on entity importance (cascade=2x, thesis=1.8x, obligation=1.6x, action=1.5x, company=1.0x, network=0.9x) multiplied by action urgency (cascade propagation=2x, thesis indicators=1.8x, rescore=1.5x, bulk refresh=0.8x) and significance level. Assigns processing tier: IMMEDIATE (>=80), STANDARD (50-79), BATCH (25-49), DEFERRED (<25).
- **L49:** Created `process_cir_queue_prioritized()` -- load-aware priority processor. Reads messages, sorts by priority score, processes IMMEDIATE first. Under high connection load (>75%), reduces batch size and defers low-priority items. Switched cron job #7 from `process_cir_queue_batch` to `process_cir_queue_prioritized`. Created `cir_propagation_priority_distribution` view.

**Priority score examples:**
| Event | Score | Tier |
|-------|-------|------|
| Megamind cascade event | 100 | IMMEDIATE |
| Thesis indicator update | 100 | IMMEDIATE |
| New interaction processing | 91 | IMMEDIATE |
| New action scoring | 75 | STANDARD |
| Company bulk refresh | 20 | DEFERRED |

**Result:** High-value strategic signals now guaranteed to process before bulk updates. System auto-throttles under load pressure.

### Loop 3 (L50-51): Self-Healing Infrastructure

**Actions:**
- **L50:** Created `cir_self_heal()` -- 6-phase automated maintenance:
  1. Cron job health check (flags jobs with >50% failure rate)
  2. Drain ghost embedding queue messages (entities that no longer exist)
  3. Process CIR dead letter queue (log and archive)
  4. Prune weak connections (strength <0.05)
  5. Clean phantom cir_state entries (raw table name keys)
  6. Fix orphaned cir_state entries (entities deleted from source tables)
- **L51:** Scheduled as weekly cron job #17 (`cir-self-heal-weekly`, Sunday midnight UTC). First run flagged 3 cron jobs with 100% failure rate (normalize-scores, daily_strategic_assessment, cir-staleness-refresh) -- all transient "connection failed" errors from sporadic Supabase connection issues, not code bugs.

**Result:** System now self-maintains weekly. Accumulated drift (phantom entries, ghost messages, dead letters, orphaned state) cleaned automatically.

### Loop 4 (L52-53): Cross-Machine Health Aggregation
