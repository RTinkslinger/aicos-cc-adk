### Root Cause Chain
1. M12 Data Enrichment filled `page_content` on 4,575 companies (0% -> 100%) and 3,528 network (0% -> 100%)
2. The embedding trigger on these tables fires on page_content UPDATE, enqueueing re-embedding jobs
3. `process_embeddings()` reads queue messages, batches them, calls `util.invoke_edge_function('embed', ...)`
4. The `embed` Edge Function is **not deployed** (M-EMB temp machine is working on this)
5. Each cron run (every 5 minutes) reads messages, fails to invoke Edge Function, messages become visible again
6. This cycle consumes connection pool slots

**What happened:** M12's page_content enrichment changed the rows, which nullified the old embeddings via the embedding trigger pipeline. The trigger enqueues new embedding jobs but since the Edge Function can't generate embeddings, the old embeddings were effectively lost. This is a critical data regression.

The droplet agents (postgres.js) hold 37-51 idle connections. Combined with cron jobs trying to process the 17K embedding queue, the 60-connection limit is regularly exceeded. This degrades ALL database operations system-wide.

### CIR Cannot Help With Embeddings Directly
CIR's role is change propagation, not embedding generation. The embedding pipeline requires:
1. Edge Function deployment (M-EMB's job)
2. OR an alternative embedding path (e.g., calling OpenAI API from a Python script on the droplet)
CIR can only monitor the queue status and report on coverage degradation.

### Recommendation
- **Immediate:** Disable the embedding cron job (jobid 1) until Edge Function is deployed. It's burning connections for zero benefit.
- **Short-term:** M-EMB should deploy the Edge Function. Once working, the 17K queue will self-drain.
- **Fallback:** If Edge Function can't be fixed, drain the queue and re-embed from the droplet using Python + OpenAI API.

## L58-59: Data Quality Propagation Chain

### The Chain: M12 update -> CIR trigger -> M5 rescore -> M6 re-index

**Verified path:**
1. M12 updates `page_content` on companies table
2. CIR trigger `cir_companies` fires (page_content IS in column list -- confirmed)
3. Trigger calls `cir_change_detected()` which looks up propagation rules
4. Rule #7: companies UPDATE -> target_action `refresh_company_connections`
5. `process_cir_queue_item()` routes to `propagate_company_change()`
6. `propagate_company_change()` updates `updated_at` on matching actions

**BROKEN LINK:** `propagate_company_change()` does NOT call `compute_user_priority_score()`. It only touches `updated_at`. The M5 rescore is not triggered by company updates. The rescore path (`rescore_related_actions`) only fires for thesis changes.

**BROKEN LINK 2:** There is no M6 re-indexing trigger. CIR propagates to actions (M5 scoring path) but does NOT trigger FTS re-indexing or IRGI function cache invalidation when page_content changes.

### Propagation Log Evidence (24h)

|-----------|-------|

| propagate_network_change | 12,307 |

| error:compute_user_priority_score(integer) | 7 |

The 7 errors from `compute_user_priority_score(integer)` confirm the M7->M5 function signature mismatch is still present (expects a row type, not integer).

### What Works
- Company update -> CIR detects it (trigger fires)
- CIR -> queue -> propagation to related actions (updated_at touched)
- Thesis update -> CIR -> rescore_related_actions -> M5 scoring
- Cascade events (M7) -> CIR -> thesis indicator update + rescore

### What's Broken
- Company update does NOT trigger M5 rescore (only updated_at)
- Company update does NOT trigger M6 re-indexing
- Network update does NOT trigger M5 rescore
- `compute_user_priority_score(integer)` signature mismatch (expects row type)
- No propagation rule for M6 IRGI functions when underlying data changes

### P1: system_health_aggregate() Misleading
- **What:** Reports grade A / 9.5 while actual intelligence quality is 3/10

- **Impact:** Intelligence functions return noise alongside signal
- **Fix:** Prune Unnamed connections, reclassify non-company entities, tighten vector similarity threshold

### P2: compute_user_priority_score(integer) Signature Error
- **What:** 7 errors in propagation log -- function expects row type, not integer
- **Impact:** Action rescoring fails silently in some CIR paths
- **Fix:** Align function signature or add wrapper
