## L68-69: Connection Pool Fix

### Problem
Previous audit found 65 backends vs 60 max_connections. Droplet agents hold 37-51 idle connections via postgres.js. Connection pool saturation causes intermittent SUPERUSER slot errors.

### Current State (at audit time)
| Metric | Value |
|--------|-------|
| Total backends | 26 (was 65) |
| Max connections | 60 |
| Utilization | 43.3% |
| Idle connections | 14 |
| Active connections | 10 |
| Idle > 5 min | 2 |
| Idle > 30 min | 2 |
| Status | HEALTHY |

The pool pressure has eased since the droplet agents aren't actively connected (0 postgres.js backends vs. 37-51 when active). The problem recurs when agents reconnect.

### Solution: Two New Functions

**`connection_pool_status()`** returns:
- Summary: backends, utilization %, headroom, idle counts by duration
- By-application breakdown: which apps hold how many connections
- Long idle connections: PIDs, app names, idle duration (candidates for termination)
- Recommendations: auto-generated based on thresholds (>90% CRITICAL, >70% WARNING)

**`terminate_idle_connections(idle_minutes INT DEFAULT 5)`** returns:
- Terminates idle connections older than threshold
- Protects: current session, supabase_admin, pg_cron, pg_net, postgres_exporter
- Returns count of terminated and skipped connections
- Safe to run on cron or manually during pool pressure

### Recommendation
Add a cron job to run `terminate_idle_connections(10)` every 15 minutes if pool pressure returns:
```sql
SELECT cron.schedule('pool_cleanup', '*/15 * * * *', $$SELECT terminate_idle_connections(10)$$);
```
Not activated yet -- wait to see if pool pressure returns when droplet agents reconnect.

## L72-73: Agent Feedback Store Setup

### Tables Created

**`agent_feedback_store`**
| Column | Type | Purpose |
|--------|------|---------|
| id | SERIAL PK | |
| action_id | INT FK -> actions_queue | Links feedback to specific action |
| agent_name | TEXT NOT NULL | 'eniac', 'datum', 'cindy', 'megamind', 'content_agent', 'sync_agent' |
| feedback_type | TEXT NOT NULL | 7 types: scoring_reasoning, routing_decision, enrichment_result, quality_assessment, propagation_decision, strategic_assessment, error_recovery, other |
| feedback_text | TEXT NOT NULL | The reasoning/feedback content |
| context | JSONB | Structured metadata (input data, confidence, etc.) |
| created_at | TIMESTAMPTZ | Auto-timestamped |

**`user_feedback_store`**
| Column | Type | Purpose |
|--------|------|---------|
| id | SERIAL PK | |
| page | TEXT NOT NULL | WebFront page: 'actions', 'strategy', 'comms', 'thesis', 'network', 'companies' |
| item_id | TEXT | ID of the rated item |
| item_type | TEXT | 'action', 'thesis', 'obligation', 'digest', 'connection', 'score' |
| feedback_type | TEXT NOT NULL | 7 types: accept, dismiss, rate_quality, flag_wrong, flag_missing, comment, preference |
| feedback_text | TEXT | Optional description |
| rating | INT 1-10 | For rate_quality type |
| context | JSONB | What was shown, session info |
| created_at | TIMESTAMPTZ | Auto-timestamped |

### Supporting Objects
- **4 indexes per table** (action_id/page, agent_name/item_type, feedback_type, created_at DESC)
- **RLS enabled** on both tables
- **RLS policies:** authenticated read+insert, service_role full access
- **`feedback_summary` view:** Aggregates agent breakdown, user breakdown, acceptance rates by item_type -- ready for `compute_user_priority_score()` to consume
- **agent_feedback_store has 2 rows** (auto-populated by other functions that log feedback)
- **user_feedback_store has 0 rows** (awaits WebFront integration)

### Self-Learning Loop Design
```
User interacts on WebFront
  -> accept/dismiss/rate/flag -> user_feedback_store
  -> feedback_summary view aggregates acceptance_rate by item_type
  -> compute_user_priority_score() reads preference_weight_adjustments
  -> preference_weight_adjustments updated from feedback_summary
  -> scores recalculated -> better recommendations next time
```

## What CIR Built (cumulative, for cross-machine context)

CIR now provides 4 monitoring dashboards callable by any machine or agent:

1. **`system_health_aggregate()`** -- Honest product quality (7 dimensions, weighted 70% product / 30% technical)
2. **`connection_pool_status()`** -- Live connection pool monitoring with per-app breakdown and recommendations
3. **`terminate_idle_connections(minutes)`** -- Safe idle connection cleanup (protects system processes)
4. **`embedding_recovery_dashboard()`** -- Embedding recovery progress with queue, coverage, processing rate, and recommendations

Plus the self-learning infrastructure:
5. **`agent_feedback_store`** table -- Agent reasoning traces (7 feedback types)
6. **`user_feedback_store`** table -- User feedback (7 feedback types + rating)
7. **`feedback_summary`** view -- Aggregated feedback for scoring model consumption
