## Bug Found: Deadlock in rescore_related_actions

### The Problem

Two cron jobs compete for row-level locks on `actions_queue`:

1. **`cir-proactive-staleness-refresh`** (schedule: `30 */4 * * *`) calls `proactive_refresh_stale_entities()` which calls `rescore_related_actions()` — does FOR loop UPDATE on `actions_queue.strategic_score`
2. **`score-refresh`** (schedule: `*/30 * * * *`) calls `refresh_active_scores()` — does FOR loop UPDATE on `actions_queue.user_priority_score`

Both run at minute 30, so they collide 6 times per day (at 00:30, 04:30, 08:30, 12:30, 16:30, 20:30). When both FOR loops lock the same rows in different orders, PostgreSQL detects the deadlock and kills one.

### Impact

Low. Only 1 deadlock in 24h out of 6 collision windows = 17% hit rate. The killed cron is automatically retried next cycle. No data corruption — PostgreSQL rollbacks are clean. No dead letters generated.

### Recommended Fix

Offset the `cir-proactive-staleness-refresh` cron by 5 minutes to avoid collision:

```sql
-- Change from "30 */4 * * *" to "35 */4 * * *"
SELECT cron.alter_job(15, schedule := '35 */4 * * *');
```

This eliminates the collision window entirely without changing functionality.
