## Issues Diagnosed & Fixed

### 1. M7->M5 Strategic Score Feedback Loop — FIXED
**Root cause:** `recalibrate_strategic_scores()` computed new strategic scores for all actions but only RETURNED them as a result set — it never wrote them back to `actions_queue.strategic_score`. The `regrade_strategic_trigger` on `actions_queue` was wired correctly but never fired because scores were never updated.

**Fix:** Created `apply_strategic_recalibration()` — wraps `recalibrate_strategic_scores()`, writes results back to `actions_queue`, logs the operation. Wired it into `process_cir_queue_item` under the `process_cascade_propagation` action so cascade events from M7 Megamind now auto-flow through to M5 scoring. Added cron job (job 21, every 6 hours at :30).

**Impact:** 140/144 actions had strategic_score updated on first run. Max delta: 5.5 points. The `regrade_strategic_trigger` now fires correctly, updating depth_grades too.

### 2. `compute_user_priority_score(integer)` Error — TRANSIENT, NO FIX NEEDED
**Root cause:** 7 errors at 2026-03-21 02:23:00, all from the same batch. The function exists with signature `(actions_queue)` and all callers pass the correct row type. Reproduced the call path successfully — no error. Likely caused by a transient schema mismatch during a migration. Not recurring.

### 3. Score Compression (stddev 1.49 vs target 2.0) — FIXED
**Root cause:** `normalize_all_scores()` (cron job 6, every 6h) applied percentile normalization across ALL actions including dismissed/done. Dismissed actions with low scores dragged the percentile distribution, compressing active actions into the 6-9 range (89.7% of active actions scored 6-9).

**Fix:** Updated `normalize_all_scores()` to normalize active and inactive actions separately. Active actions get full 1-10 spread via percentile ranking. Dismissed/done actions get compressed to 1-4 range (they shouldn't compete with active actions).

**Impact:** Active action stddev: 1.49 -> **2.66**. Max bucket: 27.1% -> **11.9%**. Score accuracy dimension: 4 (COMPRESSED) -> **10 (WELL-SPREAD)**.

### 4. `system_health_aggregate` Unnamed Connections — TYPE-AWARE BUG FIXED
**Root cause:** The unnamed connection counter queried `entity_connections` matching `source_id`/`target_id` against unnamed company IDs WITHOUT checking `source_type`/`target_type`. Since unnamed companies have low IDs (2, 3, 4, 5...) that collide with network, thesis, portfolio IDs, the query was counting cross-type collisions as "unnamed noise." 2,657 was entirely a phantom count.

**Fix:** Updated `system_health_aggregate` to use type-aware joins: checks `source_type = 'company'` before matching `source_id` against companies, and same for network. Also extended to check unnamed network entries.

**Impact:** Unnamed noise: 2,657 -> **3** (real). Connection quality score: 4.7 -> **5.3**.

### 5. Network Embedding Recovery — MONITORING
Network embeddings: 32.1% (session start) -> 36.3% (early session) -> **46.9%** (session end). Queue depth: 2,081 remaining. Recovering steadily via embedding cron (job 1, every 2 min, batch 15).

### 6. 2,657 Unnamed Entity Connections — FALSE POSITIVE
The "2,657 connections involving unnamed entities" from the prior session was a false positive caused by bug #4 above. Real count: **3 connections** involving 1 unnamed network entry. No pruning needed.
