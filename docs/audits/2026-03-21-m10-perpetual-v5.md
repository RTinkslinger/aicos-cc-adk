## 3. Key Finding: key_question_relevance Multiplier Analysis

**M12 enriched 142/142 portfolio records with key_questions. Does scoring's key_question_relevance multiplier fire more?**

YES. **66 out of 144 actions (45.8%) fire the key_question_relevance multiplier**, ranging from 1.06x to 1.16x.

| Multiplier Value | Count | Meaning |
|------------------|-------|---------|
| 1.16 | 28 | Both key_question match AND high_impact match |
| 1.10 | 24 | Key_question match only |
| 1.06 | 14 | High_impact match only |
| 1.00 | 78 | No portfolio company mentioned in action text |

**How it works:**
1. Function finds a portfolio company mentioned in the action text (40 actions mention a portfolio company by name)
2. For matched actions, it checks `portfolio.key_questions` against the action text (40% keyword overlap threshold)
3. It also checks `portfolio.high_impact` field (30% overlap threshold)
4. Max multiplier: 1.18x (both match + cap)

**Why 66 fire despite only 40 portfolio name matches:** The function also uses `company_notion_id` fallback (Strategy 2: action -> companies -> portfolio via notion IDs), catching 26 additional actions.

**Impact on scoring:** The multiplier is one of 16 multiplicative factors in `compute_user_priority_score`. A 1.16x boost on base_score of 5.0 = +0.80 points. For portfolio-linked actions, this can be the difference between bucket 5 and bucket 6.

## 9. Can CIR Detect When an Agent SHOULD Run?

**Current state:** CIR has 10 propagation rules and 11 triggers that fire on data changes across all core tables. The `cir_queue` + `process_cir_queue_item()` router dispatches to propagation functions (rescore, propagate_company_change, propagate_network_change, update_thesis_indicators). But there is NO "agent trigger" mechanism -- CIR detects changes and propagates data, but doesn't queue agent runs.

**What exists that could be extended:**
1. **`datum_requests` table** (0 rows) -- has columns for entity_type, entity_id, field_name, suggested_value, status. Designed as a work queue for Datum agent. CIR could INSERT rows here when it detects stale entities or enrichment opportunities.
2. **`cir_propagation_rules`** -- 10 rules mapping source_table/event -> target_action. Could add new rules like: `(network, INSERT) -> queue_datum_enrichment`, `(companies, UPDATE with page_content=NULL) -> queue_web_enrichment`.
3. **`cai_inbox` table** (1 row) -- existing relay for CAI messages. Could be extended to receive CIR-generated "agent should run" messages.

**Infrastructure improvements for autonomous agent life:**

| Improvement | Impact | Complexity |
|-------------|--------|------------|
