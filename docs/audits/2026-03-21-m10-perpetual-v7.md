## 6. Infrastructure Gap Assessment

### Gap 1: datum_requests Auto-Population
**Status:** Table exists with correct schema (17 columns), but **0 rows**. No auto-population trigger exists.
**Schema:** `entity_type`, `entity_id`, `field_name`, `current_value`, `suggested_value`, `suggestion_confidence`, `suggestion_source`, `status`
**Need:** A trigger or cron that detects empty/incomplete fields on network/companies and auto-creates datum_requests for the Datum agent to process.
**Priority:** P1 -- this is the bridge between automated gap detection and agent-driven enrichment.

### Gap 2: Embedding Completion Notification
**Status:** No trigger fires when embedding coverage hits 100% for a table. The `notifications` table exists (37 rows) with a generic schema (`source`, `type`, `content`, `metadata`).
**Current triggers:** `embed_*_on_insert` and `embed_*_on_update` queue embeddings, `clear_*_embedding_on_update` clears stale embeddings. But no "all done" signal.
**Need:** A function that checks coverage after each embedding batch completes and writes to `notifications` when a table crosses 95%/99%/100%.
**Priority:** P2 -- nice to have for observability.

### Gap 3: Daily Cron Auto-Retry
**Status:** `process-embeddings` runs every 2 min and has advisory locking + timeout protection. `cleanup-embedding-jobs` runs on alternating minutes to clean dead jobs. This is solid.
**Current:** The embedding processor already auto-retries via the queue pattern -- failed jobs stay in the queue and get re-picked. 279 failures in 24h were all during the 08:45 pool pressure spike and self-healed.
**Assessment:** **ALREADY IMPLEMENTED.** The `pgmq` queue + cron pattern IS the auto-retry mechanism. No additional work needed.

### Gap 4: ContentAgent Triggering
**Status:** Notification #37 (from ContentAgent) flagged that the orchestrator triggers content pipeline 5x in ~5 minutes against a source with 60-min check interval. Wasting ~$0.75/burst.
**Need:** Orchestrator should gate re-triggers using `last_pipeline_run.txt` + `check_interval_min` before firing.
**Priority:** P1 -- active cost leak.

### Gap 5: Circuit Breaker
**Status:** `check_cron_health()` function exists and runs every 6 hours. Detects >50% failure rate (CRITICAL) and >20% failure rate (WARNING). Logs to `cir_propagation_log`.
**Current behavior:** Logs health status but does NOT disable failing crons.
**Need:** When a job hits CRITICAL status (>50% failure rate), the circuit breaker should `UPDATE cron.job SET active = false WHERE jobid = X` and create a notification for human review.
**Priority:** P2 -- current health monitoring is adequate (Grade A, 0.01% error rate), but a true circuit breaker would add resilience.
