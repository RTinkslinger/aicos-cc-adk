## L76-77: Propagation Log Flood Diagnosis & Fix

### Problem
115,632 rows (35MB) in `cir_propagation_log` accumulated in ~20 hours. Root cause: a triple-log cascade:
1. Embedding processor writes back to company/network rows, triggering `cir_change_detected()` on column changes (page_content, etc.)
2. `cir_change_detected()` logs EVERY row + enqueues to CIR queue (even for `significance: 'low'` rules)
3. CIR queue processor calls `propagate_company_change()` / `propagate_network_change()` which log AGAIN even when zero actions affected

Hourly breakdown showed 25k-42k UPDATE logs during embedding processing bursts (07:00-08:00 UTC).

### Fix Applied
Three changes to break the cascade:

| Function | Change | Impact |
|----------|--------|--------|
| `cir_change_detected()` | Skip INSERT into propagation_log for `significance = 'low'` rules. Still enqueues to CIR queue. | Eliminates ~90% of trigger-generated log rows (companies UPDATE = rule 7, low significance) |
| `propagate_company_change()` | Only log when `v_affected > 0`. Increased throttle from 60s to 300s. | Eliminates all zero-impact propagation logs |
| `propagate_network_change()` | Same: only log when affected > 0, throttle 60s to 300s | Same |

### Expected Reduction
Previous: ~115k rows/day. Expected: ~5-10k rows/day (only high/medium significance events + actual propagation impacts).

Also changed log retention cron from 7 days to 2 days (`cir-log-retention`, job 11).
