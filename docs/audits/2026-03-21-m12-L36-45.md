## Embedding Pipeline Status

### Diagnosis (L36-37)

The embedding pipeline uses a pgmq queue (`embedding_jobs`) processed by an `embed` edge function called via `util.process_embeddings()` cron job every 5 minutes.

**Pipeline IS healthy and actively processing:**
- Cron job: running every 5 minutes, consistently succeeding
- Edge function: returning HTTP 200, 10 items per batch, 0 failed jobs
- Processing rate: ~3,900 items/hour (390 edge function calls/hour x 10 items each)
- Queue draining ETA: ~4.8 hours from measurement time

**Root cause of 15K backlog:** M12 L16-35 enrichment batch updated `page_content` on all companies. The `clear_companies_embedding_on_update` trigger correctly clears embeddings and `embed_companies_on_update` re-queues them. This created a bulk backlog of 14,249 company + 676 network jobs. The L40-41 network enrichment added another 4,204 network re-queue jobs.

**540 cron failures in 24 hours:** Intermittent `connection failed` errors (1 per batch occasionally). These are transient edge function cold-start/timeout issues. Not blocking -- the next run picks up the same jobs.

### New Function: `embedding_recovery_status()`

Returns comprehensive embedding pipeline status:
- Queue depth by table
- Embedding coverage percentages
- Processing rate and ETA
- Cron health and failure count
- Visibility timeout status

## Loop Details

### L36-37: Embedding Recovery Monitoring

**Created:** `embedding_recovery_status()` function
**Finding:** Pipeline healthy, processing at 3,900 items/hour. 15K backlog from L16-35 enrichment. No action needed -- queue will self-drain in ~5 hours. Interactions (23 items, 0 embedded) have 68 duplicate queue entries that will resolve naturally.

### L38-39: Company Content Quality Upgrade

**Target:** 133 portfolio companies in companies table with page_content < 500 chars
**Method:** Combined ALL available context from both `companies` and `portfolio` tables:
- Company data: sector, type, deal status, website, funding, tags, sells_to, domain, founding timeline
- Key people: resolved from `current_people_ids` via network table with names and roles
- Portfolio context: health, stage, category, entry cheque, ownership %, scale of business
- Strategic context: thesis connection, key questions, external signal, high impact notes, spikey score
- Investment notes: IDS notes (up to 800 chars), smart money, JTBD status

**Result:** 133 companies enriched. Average content length: 553 chars (was ~350). 72.5% now at 500+ chars (was 6.3%). Richest entry: 1,350 chars (AskMyGuru -- has extensive IDS notes). Zero entries under 300 chars.

### L40-41: Network Professional Context

**Created:** `enrich_network_professional_context()` function
**Method:** For each network member with company links:
- Current company name with sector tag
- Past companies with sector tags
- Portfolio connection status (company name, health, stage)
- Sourcing attribution (which portfolio companies they sourced)
- Piped deals count
- Professional context: role, location, email, RYG status, engagement playbook, leverage

**Result:** 3,069 members enriched with company context (87% of network). 322 members flagged as portfolio-connected. 459 members without company links enriched with available metadata (role, location, relationship tags).

### L42-43: Data Freshness Scoring

**Created:** `entity_freshness_score(entity_type, entity_id)` function
- Composite score 0-100 based on: update recency (30%), Notion edit (25%), sync (20%), enrichment (15%), connection activity (10%)
- Grades: A (90+), B (70-89), C (50-69), D (30-49), F (<30)
- Includes action mention check for companies

**Created:** `stale_entities_report(days_threshold)` function
- Summary counts of stale entities
- Priority list: stale portfolio companies first, then pipeline companies
- Freshness distribution: fresh (7d), recent (30d), aging (90d), stale (90d+)

**Current state:** Zero stale entities (all touched by enrichment). Function will become critical as time passes for M10 CIR's proactive refresh triggers.
