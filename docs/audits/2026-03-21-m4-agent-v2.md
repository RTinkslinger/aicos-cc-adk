## What Was Done

**Before:** 956 lines. Contained step-by-step processing loops, copy-pasted SQL patterns, decision trees embedded as scripts, and duplicated content from skills. The agent was being told HOW to do everything — undermining the ClaudeSDKClient pattern where the agent reasons autonomously.

## Agent Build Review

### Is this objective-driven?

**YES.** The CLAUDE.md defines 6 clear objectives, not processing scripts. The agent knows WHAT to achieve (entity integrity, identity resolution, enrichment, staging pipeline, graph maintenance, quality) and has BOUNDARIES (confidence gating, database access rules, batch limits). It does not have step-by-step instructions for how to process each message type.

### Are skills rich reference material?

**YES.** 6 skills totaling 1,622 lines of reference material:

| Skill | Lines | Quality Assessment |
|-------|-------|-------------------|
| `datum-processing.md` | 198 | Good — input parsing patterns, field mapping, column names, schema reference, processing checklist |
| `dedup-algorithm.md` | 275 | Excellent — 4-tier dedup for persons and companies, merge protocol, COALESCE pattern, decision flowchart, special cases |
| `people-linking.md` | 252 | Excellent — 6-tier resolution with SQL examples, cross-surface stitching, surface-specific flows, edge cases, confidence thresholds |

Skills include:
- Tool signatures with WHEN to use
- Patterns of success (SQL examples, interpretation guides)
- Anti-patterns (what NOT to do)
- Decision frameworks (flowcharts, matrices)
- Edge cases and special handling

### Would this impress the Agent SDK team?

**Assessment: YES.** The architecture follows the definitive pattern:

1. **Python = lifecycle only.** Hooks handle iteration logging, compaction, and token checks. No intelligence in Python.
2. **CLAUDE.md = objectives.** 231 lines of identity, goals, boundaries, and anti-patterns. No processing scripts.
3. **Skills = rich reference.** 1,622 lines of domain knowledge loaded on demand based on task type.
4. **Agent REASONS.** Given objectives and boundaries, the agent decides how to chain tools. Skills inform but don't prescribe.
5. **All tools allowed.** Bash, Read, Write, Edit, Grep, Glob, Agent, Skill — full autonomy.
6. **permission_mode="dontAsk".** Fully autonomous operation.
7. **Production primitives used.** PreCompact hook, session checkpoint, Stop hook for iteration tracking, UserPromptSubmit hook for compaction signals.

## Remaining Work

1. **Skills directory location:** Skills are at `mcp-servers/agents/skills/datum/` (shared skills directory). The CLAUDE.md references them as `skills/datum/...`. On the droplet the cwd is `/opt/agents/datum/` and skills are at `/opt/agents/skills/datum/`. Verify the Skill tool resolves these paths correctly at deploy time.

2. **Interaction staging source-specific processing:** The detailed per-source processing instructions (email, WhatsApp, Granola, Calendar) were removed from CLAUDE.md. They exist partially in `people-linking.md` (resolution flows by surface). Consider whether a dedicated `interaction-staging.md` skill would be valuable to consolidate the staging processing patterns that were in CLAUDE.md section 3b.

3. **Web enrichment strategy:** The decision tree for web enrichment (LinkedIn first, then search, then domain scrape) was removed from CLAUDE.md. It's implied by the enrichment priorities + anti-patterns + web tools listing, but a dedicated `web-enrichment.md` skill could make this more explicit if the agent struggles with enrichment decisions.
