## L36-37: EMERGENCY -- Preference Weight Reset + Guard

### Problem (P0 from M9)
The `preference_weight_adjustments` table had 27 rows with systematically anti-user weights:
- Pipeline Action: **-0.40** (12 samples)
- Pipeline/Deals: **-0.40** (4 samples)
- Meeting/Outreach: **-0.33** (24 samples)
- Cindy-Meeting source: **-0.40** (6 samples)
- P0 priority: **-0.34** (14 samples) -- penalizing the HIGHEST priority
- Thesis Research: **-0.40** (33 samples)

**Root cause:** 64 total decisions but only 16 accepted vs 48 dismissed (75% dismiss rate). The learning function required only 2 samples per dimension, so it wrote weights based on noise. Since most actions were dismissed (bulk cleanup), the system "learned" to penalize everything.

### Fix
1. **DELETE all 27 rows** from preference_weight_adjustments
2. **Rewrote `update_preference_weights()`** with triple guard:
   - Global minimum: 100 total accept/dismiss decisions (was 0)
   - Global accept rate: must be >= 20% (new guard)
   - Per-dimension minimum: 15 samples with >= 3 accepts each (was 2 samples, no accept requirement)
3. **Verified guard blocks:** Function now returns early when called (only 64 decisions, below 100 threshold)

### Impact
Top-10 ranking immediately improved -- Pipeline/Deals returned to #1-2, Cindy-Meeting actions properly represented.

## L38-39: Action Deduplication

### Problem (P2 from M9)
~20% action dedup contamination. Cultured Computers investment decision appeared twice in top-5.

### Analysis
Found 4 candidate pairs using trigram similarity (> 0.40):
1. Cultured Computers investment decision (ids 118 vs 132, sim=0.677) -- TRUE duplicate
2. AuraML endorsement (ids 119 vs 133, sim=0.566) -- TRUE duplicate
3. Case study request (ids 27 vs 37, sim=0.556) -- TRUE duplicate (generic vs specific)
4. Funding discrepancy (ids 67 vs 71, sim=0.419) -- FALSE positive (different companies)

### Fix
Created `deduplicate_actions(threshold, dry_run)` function:
- For each pair above threshold, keeps the higher-scored action
- Dismisses the lower-scored one with `outcome = 'Auto-dismissed: duplicate of #X (similarity=Y)'`
- Guards against cascading dismissals within a single run
- Default threshold: 0.55

### Result
3 duplicate pairs resolved. Actions 132, 133, 27 dismissed. Top-10 clean.

## L40-41: Interaction Recency Integration Fix

### Problem
Only 10.1% of actions received interaction recency boost (M9 found 12.5%). Target: >40%.

### Root Cause Analysis
The `interaction_recency_boost()` function had 2 matching strategies:
1. Direct name match (participant/people/company names in action text)
2. company_notion_id lookup

**Issues:**
- Strategy 2 never fired (0 actions have company_notion_id)
- Strategy 1 failed when action text didn't contain exact participant/company name
- Boost window was 21 days but scan window was 30 days (Levocred at 23 days found but returned 0)

### Fix: Added 3 new strategies + extended window
3. **Portfolio company name match** -- portfolio table names -> companies -> interactions
4. **Thesis connection graph** -- thesis_connection -> thesis_threads -> entity_connections -> company -> interactions
5. **Summary co-occurrence** -- company name appears in both action text and interaction summary

Extended boost decay from 21 to 30 days (`exp(-0.10 * days)` from `exp(-0.12 * days)`).

### Result
| Metric | Before | After |
|--------|--------|-------|
| Actions with boost | 7 (10.1%) | 26 (53.1%) |
| Avg boost value | 0.198 | 0.197 |
| Max boost | 0.788 | 0.788 |

Levocred, Supermemory, MSC Fund now correctly receive boosts.
