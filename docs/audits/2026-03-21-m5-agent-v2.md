## 4. CRITICAL FINDINGS

### Finding 1: Score History Snapshots NOT on Cron (MEDIUM)

**Problem:** `snapshot_scores()` is NOT called by any cron job. The function exists inside `refresh_action_scores()` (the FULL refresh), but the cron job calls `refresh_active_scores()` (the lightweight drift-check), which does NOT snapshot.

**Evidence:**
- No cron job matches `%snapshot%`
- Last snapshot: 2026-03-21 10:58 UTC (580+ minutes ago, 9.7 hours stale)
- `refresh_active_scores()` source code: only updates scores, no snapshot call
- `refresh_action_scores()` does call `PERFORM snapshot_scores()` but is NOT on cron

**Impact:** `score_trend()` and `scoring_velocity()` functions return stale data. Trend analysis is degraded. Score history has only 227 records total instead of growing.

**Fix Required:** Either:
- (a) Add `snapshot_scores()` call at end of `refresh_active_scores()`, or
- (b) Add a separate cron: `SELECT snapshot_scores()` every 30 min

### Finding 2: 18/41 Active Actions Not Context-Enriched (MEDIUM)

**Problem:** 43.9% of proposed/accepted actions have not been enriched by `enrich_action_context()`. This means they have default confidence (0.5) and their scoring_factors lack company_name, company_context, person_name, thesis_context, and source_summary.

**Breakdown of 18 non-enriched actions:**
- 8 obligation_followup actions (IDs 139-146): ALL scoring_factors NULL for core fields (time_sensitivity, conviction, etc.)
- 10 original actions (IDs 14, 29, 32, 55, 78, 92, 95, 99, 100, 113): Have scoring_factors but no context_enriched flag

**Impact:** Confidence test fails (8 actions). Score quality degraded for these actions because multipliers that depend on enriched context (company_context, person_name) cannot fire.

**Fix Required:** Run `SELECT enrich_action_context(id) FROM actions_queue WHERE status IN ('Proposed','Accepted') AND NOT COALESCE((scoring_factors->>'context_enriched')::boolean, false)`

### Finding 3: P0 Actions Scoring Below 7.0 -- Root Cause Identified (LOW-MEDIUM)

**Two P0 actions (IDs 136, 137) score 6.25 and 6.37 despite being P0 priority.**

**Root cause: Low base scores (4.64 and 4.72).**
- These are obligation/operational actions with ALL non-strategic factors defaulting to 0.5
- After z-score normalization, factors like time_sensitivity (0.219), conviction_change (0.345), stakeholder_impact (0.176) are below mean
- The only strong factor is strategic_score (0.803/0.881) which carries 20% weight
- Combined multiplier is capped at 1.35 for both (maxed out -- they have many boosts but can't exceed cap)
- The base score is so low (4.6-4.7) that even 1.35x boost only reaches 6.2-6.4

**Why this is a design tension, not a bug:**
- The sigmoid wall at 8.0 + cap at 1.35 prevents these from reaching 7.0+
- The model correctly penalizes actions with generic/default scoring_factors
- But P0 actions from trusted sources (whatsapp/Cindy) probably DO deserve higher scores

**This is the eternal tension: should priority label override model quality?** Currently the model says "I don't have enough data to justify a high score" even when P0. The priority multiplier (1.15) is included but insufficient when base factors are all defaults.

## 5. Skill Files Alignment Assessment

### Alignment with Persistent Agent Pattern

The scoring skills define **objectives** (what agents should know and do) rather than **scripts** (step-by-step instructions). This aligns well with the persistent agent pattern where agents are autonomous and self-directing.

| Aspect | Alignment | Notes |
|--------|-----------|-------|
| Objective-driven | GOOD | Skills describe WHAT to check, not prescriptive steps |
| Agent autonomy | GOOD | Each agent can decide when/how to use scoring context |
| Feedback loop | GOOD | Clear protocol for agents to influence scores |
| Self-improving | GOOD | Preference learning + verb pattern learning are self-adjusting |
| Cross-agent | GOOD | All 5 agents + orchestrator can read and write feedback |

### Skill Gaps Identified

**Gap 1: No skill for "when to trigger enrichment"**
Agents don't know that 43.9% of actions lack enrichment. No skill teaches agents to detect and fix enrichment gaps. Datum agent should own this.

**Gap 2: No skill for "obligation action scoring"**
8 obligation-generated actions have no scoring_factors at all. The scoring model defaults everything to 0.5 for them. No skill teaches how obligation actions should be scored differently from content-agent-generated actions.

**Gap 3: Skills reference v5.1-L96 model version**
All 3 skill files reference v5.2-L96 in the header but the internal SQL examples show v5.1-L96 model_version strings. This is because the live functions still return v5.1-L96. Should be updated together.

**Gap 4: No skill for score history / trend monitoring**
Agents need to know they can use `score_trend(action_id)` and `scoring_velocity()` for temporal analysis, but these tools are degraded because snapshots aren't on cron (Finding 1). Once fixed, a monitoring skill would help agents detect score instability.

| Type | Avg Score |
|------|-----------|
| Portfolio avg | 8.46 (Accepted only) |
| Pipeline avg | 7.15 |
| Thesis avg | 6.89 |

### What New Tools Would Agents Need?

1. **`enrichment_status()` function** -- Returns per-action enrichment gaps (what's missing, what could be enriched). Datum agent calls this to decide what to enrich next.

2. **`obligation_scoring_factors(action_id)` function** -- Generates scoring_factors for obligation-generated actions by analyzing the obligation itself (person priority, overdue duration, promise type). Currently these get all-0.5 defaults.

3. **`score_explanation_diff(action_id1, action_id2)` function** -- "Why does action A score higher than action B?" Returns the top 3 multiplier/factor differences. Useful for Megamind when reasoning about relative priorities.

4. **`model_calibration_report()` function** -- Returns: factor-score correlations, multiplier effectiveness (which multipliers actually differentiate accepted vs dismissed), dead multipliers (always 1.0), and suggested weight adjustments. Currently the model evolves by manual analysis -- this would enable data-driven evolution.
