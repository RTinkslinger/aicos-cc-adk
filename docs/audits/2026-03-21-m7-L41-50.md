## Loop Details

### L41-42: Cascade Duplicate Guard

**Problem:** `process_obligation_cascade()` re-processes same obligations on every run, re-boosting scores. `auto_generate_obligation_followup_actions()` uses brittle ILIKE matching (first 40 chars) that misses near-duplicates.

**Specialist 1 (Analyst):** Traced the two duplication vectors: (1) cascade re-boosts already-maxed scores, (2) followup generator creates actions textually similar to existing ones but different enough to bypass ILIKE.

**Specialist 2 (Strategist):** Designed `cascade_dedup_guard(action_text, threshold)` using pg_trgm similarity. Default threshold 0.6. Returns `is_duplicate`, `matching_action_id`, `similarity_score`.

**Specialist 3 (Engineer):** Built three changes:
1. **`cascade_dedup_guard()`** -- trigram-based duplicate detection, returns matching action details
2. **`process_obligation_cascade()`** -- rebuilt with: skip Dismissed/Done actions, diminishing returns (half boost if cascaded within 7 days), skip already-maxed scores (>= 9.5)
3. **`auto_generate_obligation_followup_actions()`** -- replaced ILIKE dedup with `cascade_dedup_guard()` at 0.4 threshold

**Specialist 4 (QA):** Verified dedup guard catches near-duplicates (0.82 similarity on MSC Fund text) and correctly allows novel actions (0 similarity on unrelated text). Both false positive and false negative tests passed.

### L47-48: Strategic Network Analysis

**Problem:** No automated way to identify the most strategically important people in the 3,528-person network.

**Specialist 1 (Analyst):** Initial implementation using first-name ILIKE matching caused massive false positives -- "Rajat" from obligations matched ALL 10+ Rajat records in network. Root cause: obligations table has person_name text AND person_id integer.

**Specialist 2 (Strategist):** Redesigned matching strategy:
- Primary: Use `person_id` joins (obligations.person_id -> network.id) for exact matches
- Secondary: Full name ILIKE matching only for action text mentions (minimum 8 chars to avoid false positives)
- Entity connections: source_type='network' with connection_type IN ('current_employee','affiliated_with','interaction_linked')

**Specialist 3 (Engineer):** Built `strategic_network_map(limit)` with CTE architecture:
1. Pre-compute obligation counts per person_id
2. Score each network person: portfolio connections (0-3), obligations (0-2.5), deal involvement (0-2), interaction recency (0-1.5), entity connection strength (0-1)
3. Filter to people with any strategic signal (portfolio, obligations, deals, or recent interactions)

**Specialist 4 (QA):** Rebuilt function eliminated all false positives. Results now correctly show:
- Mohit Gupta (3.93): 5 obligations, 3 overdue (two network records, correctly differentiated by person_id)
- Ayush Sharma (3.91): 3 obligations, all overdue
