## L58-60: Portfolio-Aware Priority Scoring

### Problem
- `recalculate_obligation_priority()` had no awareness of portfolio context
- An obligation about a company where Aakash owns 6.7% with a $360K cheque got the same treatment as a casual meeting follow-up
- SPR companies (follow-on priority) were not boosted
- Fumes dates (running out of cash) were not factored in

### Solution
Rewrote `recalculate_obligation_priority()` with portfolio context matching:

**Portfolio matching strategy:**
1. Primary: `network.current_company_ids` -> `portfolio.company_name_id` (UUID-based, reliable)
2. Fallback: obligation `person_role` field parsed for company name -> `portfolio.portfolio_co` ILIKE match

**Portfolio boost factors (additive, capped at 1.0):**
| Factor | Threshold | Boost |
|--------|-----------|-------|
| Ownership >= 3% | high_ownership | +0.15 |
| Ownership >= 1.5% | mid_ownership | +0.10 |
| Ownership >= 0.5% | std_ownership | +0.05 |
| Entry cheque >= $200K | large_cheque | +0.10 |
| Entry cheque >= $100K | mid_cheque | +0.05 |
| Fumes date <= 3 months | FUMES! | +0.25 |
| Fumes date <= 6 months | fumes_warning | +0.15 |
| Key questions present | key_q | +0.05 |
| follow_on_decision = 'SPR' | SPR | +0.10 |
| Health = 'Red' | RED | +0.08 |

**New status transitions:**
- `pending` with past `due_date` -> automatically moved to `overdue`
- Stores `portfolio_boost`, `portfolio_reason`, and `recalc_reason` in `cindy_priority_factors` JSONB

## Key Decisions

1. **Deleted ALL 24 Cindy-Meeting obligations** rather than selectively deduplicating. Rationale: every single one was orphaned (referencing deleted interactions), and the granola source had better versions with proper person_ids, roles, and due dates.

2. **Portfolio matching via `current_company_ids`** rather than fuzzy name matching. The `network.current_company_ids` array contains Notion page UUIDs that directly match `portfolio.company_name_id`. This is deterministic and won't break on company name variations.

3. **Used `FOUND` flag** instead of `v_record IS NOT NULL` for PL/pgSQL record checks after `SELECT INTO`. The `IS NOT NULL` check on a record with many nullable columns is unreliable -- it returns TRUE only if ALL fields are non-NULL.

4. **Portfolio boost is additive** (not multiplicative) and capped at 1.0. Multiple factors compound: a Red-health SPR company with high ownership could get +0.15 + 0.10 + 0.10 + 0.08 = +0.43 boost. This is intentional -- these are the companies that matter most.
