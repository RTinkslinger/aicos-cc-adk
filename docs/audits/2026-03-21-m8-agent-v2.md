# M8 Cindy Agent v2 Audit -- CLAUDE.md Rewrite for ClaudeSDKClient Pattern

**Date:** 2026-03-21 | **Machine:** M8 | **Status:** COMPLETE

## Problem Statement

Cindy's CLAUDE.md contained a 160-line "Processing Loop" + "Core Query Patterns" section
(lines 110-270) that scripted exact SQL queries as step-by-step instructions. This violates
the persistent agent SDK pattern: **CLAUDE.md = objectives, not scripts.** A ClaudeSDKClient
agent that loads this file should understand WHAT to achieve, not be walked through exact
SQL statements like a puppet.

Reference: `docs/research/2026-03-21-persistent-agent-sdk-pattern.md`

## What Was Changed

### 1. Processing Loop Rewrite (lines 110-138 -> 110-132)

**Before:** 28 lines of exact SQL queries telling the agent what to SELECT, what columns to
UPDATE, with hardcoded query patterns.

**After:** 22 lines of objective-driven instructions:
- Primary Objective: "Extract intelligence from every unprocessed interaction"
- 9 numbered objectives (detect obligations, extract actions, surface thesis signals, etc.)
- Secondary Objective: "Keep the intelligence layer current"
- Single pointer to psql + schema skill for HOW

### 2. Core Query Patterns Removed (lines 140-270)

**Before:** 130 lines of exact psql commands for every operation: insert interaction, resolve
person by email/phone/name, link person to interaction, create context gap, fill context gap,
write action item, write thesis signal, write datum_person request, write notification,
query recent interactions, query open actions, query thesis connections.

**After:** Removed entirely. These were teaching the agent SQL syntax, not objectives. The
agent has psql, the schema skill (`skills/data/postgres-schema.md`), and 33 SQL intelligence
functions. It can construct queries from schema knowledge.

### 3. Signal Extraction Section Cleaned (Section 7)

**Before:** Each signal type (action items, thesis signals, relationship signals, deal signals)
had inline SQL INSERT examples showing exact column values.

**After:** Objective descriptions -- what to extract, where to write, what metadata to include.
Added `cindy_deal_velocity()` reference for portfolio velocity tracking.

### 4. Obligation Auto-Fulfillment Cleaned (Section 7.5)

**Before:** Inline SQL UPDATE for auto-fulfillment with exact column assignments.

**After:** Description of the behavior + pointer to `skills/cindy/obligation-reasoning.md`
for full auto-fulfillment rules.

### 5. Fleet Agent Communication Cleaned (Section 8)

**Before:** Inline SQL INSERT for datum_person inbox message with example values.

**After:** Objective description -- what to include in datum_person messages.

### 6. Pre-Meeting Context Assembly Rewritten (Section 9)

**Before:** Pseudocode loop with inline SELECT queries.

**After:** Objective-driven description pointing to `cindy_person_intelligence()` and
`person_communication_profile()` functions, with clear brief structure.

### 7. Email Architecture Corrected

Fixed `cindy@agent.aicos.ai` -> `cindy.aacash@agentmail.to` (per v6 audit correction).
Added explicit note: "Cindy does NOT scan Aakash's personal email (Gmail or Outlook)."

### 8. Obligation Dedup Anti-Pattern Added (User Feedback)

Added "Obligation Dedup (CRITICAL -- user feedback)" section after "Do NOT Create Obligations
For" list. Requires semantic comparison of existing obligations for the same person before
creating new ones. Addresses user feedback: "In my you owe section I see Ayush thrice.
Two of those tasks are pretty much the same."

## Agent Build Review

**Question: Would an autonomous ClaudeSDKClient agent know what to do reading this CLAUDE.md?**

### Strengths (retained)
- Clear identity (Section 1): who Cindy is, what she's NOT, her relationship to the fleet
- Table access matrix (Section 3): precise read/write boundaries per table
- 4 surface pipelines (Section 4): clear pipeline descriptions per surface
- Resolution algorithm (Section 5): tiered confidence approach
- Gap detection (Section 6): richness scoring formula with thresholds
- Signal routing table (Section 7): clear threshold + route decisions
- 20 anti-patterns (Section 13): comprehensive NEVER-do list
- Error handling (Section 14): per-error-type responses
- 33 SQL functions (Section 15): full registry with argument types
- Skills loading strategy (Section 17): task-to-skill mapping

### Strengths (new)
- Processing objectives (Section 3): agent knows WHAT to achieve, not WHAT to type
- Obligation dedup (Section 7.5): addresses real user pain
- Email architecture: unambiguous AgentMail reference

### Remaining Concerns
1. **Document length (1004 lines)**: Still substantial. An agent with context limits may
   struggle to hold the full document. But the skill loading strategy (Section 17)
   mitigates this -- agent loads skills on demand, not all at once.
2. **Section 4 pipelines**: Still somewhat prescriptive (8-step pipeline per surface).
   These are closer to "strategy" than "script" though -- they describe WHAT processing
   happens, not exact SQL. Acceptable for now.
3. **Section 5 resolution algorithm**: Contains pseudo-code with SELECT patterns. This is
   algorithmic description (the resolution STRATEGY), not a script. The agent needs to
   understand the tiered approach. Acceptable.
4. **Section 15 still has 4 psql examples**: These teach the calling convention for
   different argument types (no-arg, integer, jsonb, text). An agent unfamiliar with
   psql function calling patterns genuinely needs these. Acceptable.

### Verdict

**PASS.** An autonomous ClaudeSDKClient reading this CLAUDE.md would:
1. Know WHO it is and what it's NOT (observer, not actor)
2. Know WHAT to achieve (9 processing objectives)
3. Know WHERE to read and write (table access matrix)
4. Know HOW to resolve people (tiered algorithm)
5. Know WHEN to route signals (routing criteria table)
6. Know WHAT NOT to do (20 anti-patterns)
7. Know HOW to load domain knowledge (skill loading strategy)
8. Know HOW to call functions (psql convention examples)

The agent would NOT be walking through a scripted SQL tutorial. It would reason about
objectives and use its tools (psql, skills, schema knowledge) to achieve them.
