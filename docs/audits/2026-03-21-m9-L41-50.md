### The 23 INTELLIGENCE_IN_SQL Functions (Violations)

These functions try to BE the intelligence instead of providing context for agent reasoning:

1. **`detect_opportunities`** — SQL finds "cross-thesis companies" and "relationship hotspots" using CASE statements and hardcoded thresholds. An agent should look at the data and reason about opportunities.
2. **`detect_emerging_signals`** — SQL computes "spike ratios" by comparing recent vs baseline activity counts. Pattern detection should be agent reasoning.
3. **`detect_thesis_bias`** — SQL counts evidence_for vs evidence_against and flags "confirmation bias" if ratio > 5x. Bias detection requires nuance an agent provides.
4. **`predict_next_actions`** — SQL generates action suggestions from hardcoded templates ("Research contrarian evidence for..."). Action prediction = pure reasoning.
5. **`suggest_next_actions`** — SQL generates "Deep dive: Review X" and "Connect: Y" from templates. Template-based suggestions are not intelligence.
6. **`generate_strategic_briefing`** — Assembles JSON report with hardcoded structure. Should be agent-composed narrative.
7. **`generate_strategic_assessment`** — 11,414 chars of SQL that computes "convergence ratios" and "underserved theses". Strategic assessment = agent work.
8. **`generate_decision_framework`** — Builds a decision framework from templates. Decision framing = agent reasoning.
9. **`simulate_convergence`** — Models accept/dismiss outcomes. Simulation = agent work.
10. **`preference_insights`** — Analyzes accept/dismiss patterns to surface user preferences. Pattern analysis = agent work.
11. **`portfolio_risk_assessment`** — 11,487 chars of SQL computing risk scores with hardcoded weights. Risk assessment = agent reasoning.
12. **`portfolio_intelligence_report`** / **`portfolio_intelligence_map`** — SQL generates "intelligence reports". Reports = agent work.
13. **`deal_pipeline_intelligence`** — Assembles deal context as a JSON blob. Intelligence = agent reasoning.
14. **`competitive_landscape`** — Finds similar companies by embedding + shared theses. Discovery = OK, but interpreting competitiveness = agent.
15. **`detect_interaction_patterns`** / **`detect_obligation_fulfillment_from_interactions`** — Pattern detection in SQL.
16. **`strategic_network_map`** — SQL computes "influence scores" and network topology. Strategic interpretation = agent.
17. **`intelligence_timeline`** — SQL assembles timeline. Interpretation = agent.
18. **`predict_staleness_candidates`** — SQL predicts what will go stale. Prediction = agent.

### HONEST RATIO: 23 INTELLIGENCE functions / 13 AGENT_TOOL functions = **1.77:1 wrong direction**

The system has nearly **twice as many functions trying to BE intelligence** as functions serving as tools for agents. The fundamental rule says "Claude Agent SDK agents do ALL thinking, SQL = plumbing ONLY." The current ratio is inverted.

### What Scoring SHOULD Look Like

The `compute_user_priority_score` function is **435+ lines of SQL** with 15 multipliers, hardcoded weights, CASE statement trees, and ILIKE pattern matching. This is a mathematical model, which is arguably OK in SQL — but the *interpretation* of what those scores mean, which actions to surface, and how to explain them to the user should be agent reasoning.

The `agent_scoring_context` function (12,004 chars) is a genuine agent tool — it provides rich context including score explanation, multipliers, portfolio context, network context, thesis connections, interactions, and obligations. This is the RIGHT pattern.

**VERDICT: The system has built the plumbing but is trying to make the plumbing do the thinking. 23 functions need to be converted from "generate intelligence" to "provide context for agents to generate intelligence."**

### The Fundamental Problem

The system has 23 SQL functions trying to generate intelligence (detect opportunities, predict actions, assess risk, generate briefings). The new architectural rule says agents should do this reasoning. But there's **no infrastructure for storing agent-produced intelligence**. No `agent_assessments` table, no `company_intelligence` table, no `meeting_briefs` table. The system was designed for SQL to be the brain, and switching to agent-as-brain requires a storage layer for agent outputs.

### TOP 5 THINGS THAT WOULD MOST IMPROVE USER EXPERIENCE

1. **Populate key_questions for all active portfolio companies (136 remaining).** This is the highest-ROI field in the entire database. When Aakash opens a portfolio company before a call, key_questions tells him what to ask. Currently 90% empty. An agent could generate initial key_questions from existing data (stage, health, thesis connections, recent interactions) and Aakash can refine them.

2. **Build an agent-generated "meeting brief" for upcoming interactions.** Before any meeting, an agent should compose a 200-word brief: who they are, last interaction, obligations (I owe them / they owe me), relevant thesis threads, key questions, suggested talking points. Currently the user gets raw SQL query results.

3. **Fix network table embedding coverage (10.5% -> aim for 80%+).** 3,158 people are invisible to semantic search. When Aakash asks "who do I know at Sequoia?" or "who's an expert in AI infrastructure?", 89.5% of his network doesn't show up.

4. **Replace SQL intelligence functions with agent tool functions.** The 23 INTELLIGENCE_IN_SQL functions should become data-providing tools that agents call, reason about, and produce human-readable intelligence. Start with the 3 highest-value: `generate_strategic_briefing` -> agent composes briefing; `portfolio_risk_assessment` -> agent assesses risk; `predict_next_actions` -> agent suggests actions.

5. **Add an agent-intelligence storage layer.** Create tables for: `agent_assessments` (per-entity strategic assessments), `meeting_briefs` (pre-meeting intelligence packets), `company_intelligence_notes` (ongoing agent observations). The WebFront should display these agent-written artifacts, not raw SQL output.
