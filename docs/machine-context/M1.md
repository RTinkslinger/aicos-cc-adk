# M1 WebFront — Machine Context
*Accumulated across 7 loops. Product leadership reads at start, writes at end.*

## Product Vision Context
digest.wiki is Aakash's primary interface to the AI CoS system. Mobile-first. It renders agent output — agents reason, WebFront displays. Every page should answer "what should I do?" not just "here is data." The user lives on WhatsApp and mobile; desktop is secondary.

**WebFront reads agent output tables with SIMPLE SELECT. Never calls intelligence-generating SQL functions. Light SQL views can format (sort, paginate, join) but never generate intelligence.** The intelligence layer lives in agents (Cindy, Megamind, Datum, ENIAC). WebFront is a rendering surface for their output — not a consumer of complex SQL RPCs that encode reasoning.

User cares about: actionable intelligence on every page, fast mobile UX, correct data, suggestive actions (not raw data dumps), deep research accessibility, obligation tracking with EA-quality responses. Key user signal: "dumb dump" is the recurring complaint (FB-29, FB-30, FB-35, FB-36) — data without suggested actions is worthless.

## Patterns of Success
- **Deploy every loop** — user is sole consumer, needs to live-test immediately. 30+ deploys across sessions.
- **Lazy-load heavy content** — DeepResearchPanel, PersonIntelligencePanel use fetch-on-click, not page-load.
- **Null guards on all RPC responses** — every query function has try/catch returning safe defaults ([], null, 0). 100% query layer safety.
- **Optimistic UI for actions** — dismiss/accept removes immediately, persists in background.
- **Portfolio guardrails** — portfolio companies filtered from deal flows, relabeled from "Deal" to "Portfolio".
- **Conditional section rendering** — hide redundant sections (Key Contacts when Founders showing).
- **Scroll-lock pattern** — `document.body.style.overflow = "hidden"` on overlay open, restore on close. Copied from AddSignal.tsx which works correctly.

## Anti-Patterns (Learned)
- **Overlay scroll bleed** (FB-37): Panel opens but background scrolls too. Must set body overflow hidden.
- **z-index conflicts**: z-50 overlays can be hidden behind TopBar. Use z-[60]+ for slide-out panels.
- **`grid-cols-4` on mobile**: Breaks layout. Always test at 375px.
- **Using `led_by_ids` for founders** (FB-22): This field contains deal leads + co-investors. Use `portfolio_founders()` RPC.
- **ILIKE for thesis/company matching** (FB-25): Too noisy, false positives (e.g., healthcare thesis linked to irrelevant companies). Use entity_connections + semantic vector similarity.
- **Returning ALL obligations including dismissed** (FB-32): After revalidation, dismissed cards reappear. Filter at DB query level.
- **Fixed/generic action options** (FB-33): "Dismiss/Reschedule/Follow-up" regardless of context. Options should be generated per-obligation using person/company/obligation context.
- **Raw data dumps** (FB-29, FB-36): Interaction history as chronological list without intelligence overlay. Every detail view needs "what should I do about this?"
- **`interaction_patterns.channels.length` without null guard** (FB-23): RPC returns null when no interactions exist.

## Architecture Overhaul: SQL RPCs → Agent Output Tables

### Current State (PROBLEM)
The /comms page has 7 sections, each calling complex SQL RPCs directly from the frontend:
- `cindy_daily_briefing_v3()` — generates a briefing (intelligence generation in SQL)
- `cindy_relationship_intelligence()` — generates relationship analysis (intelligence in SQL)
- `cindy_person_intelligence()` — generates person intelligence summaries (intelligence in SQL)
- `portfolio_internal_intel()` — generates internal intelligence (intelligence in SQL)
- `cindy_companies_needing_attention()` — generates attention priorities (intelligence in SQL)
- `cindy_relationship_momentum()` — generates momentum analysis (intelligence in SQL)
- Various obligation RPCs with embedded priority logic

These SQL functions contain reasoning logic (scoring, prioritization, narrative generation) that should live in agents.

### Target State (CORRECT ARCHITECTURE)
Each section reads from **agent output tables** that Cindy (and other agents) write to:
- Cindy agent runs periodically → reasons about relationships, obligations, priorities → writes structured output to agent output tables (e.g., `cindy_briefings`, `cindy_relationship_intel`, `cindy_attention_alerts`)
- WebFront does `SELECT * FROM cindy_briefings WHERE date = today ORDER BY priority DESC` — simple read, no intelligence generation
- Light SQL views can join, sort, paginate, filter — but never generate intelligence
- Agent output tables have timestamps so WebFront shows freshness ("updated 5 min ago")

### Migration Path
1. Identify which SQL RPCs generate intelligence vs. which are simple data access
2. For each intelligence-generating RPC: move the reasoning to the Cindy/Megamind agent, have the agent write results to an output table
3. Replace frontend RPC calls with simple SELECT from output tables
4. Keep utility SQL (joins, filters, pagination) as lightweight views
