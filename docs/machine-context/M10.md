## Product Vision Context

M10 is the **nervous system** of the AI CoS. It ensures that every entity in the system has fresh embeddings, meaningful connections, healthy crons, and propagated intelligence. Without M10, search degrades (M6), scoring uses stale data (M5), and agents reason over garbage.

**What the user cares about:** When Aakash searches for a company or person, results must be relevant and fresh. When the scoring model ranks actions, entity connections must reflect real relationships, not auto-generated noise. When crons fire, they must not deadlock or silently fail. M10 makes all of this invisible -- it works when you never notice it.

**The nervous system metaphor:** Embeddings = sensory neurons (how entities are perceived). Connections = synapses (how entities relate). Crons = heartbeat (keeps everything alive). Propagation = signal transmission (changes ripple across the system). M10 maintains all four.

## Patterns of Success

1. **Advisory locks prevent deadlocks.** `pg_try_advisory_lock` on cron functions that touch shared tables (actions_queue, entity_connections) is the reliable pattern. Tested: zero deadlocks since L2 fix.
2. **Cron stagger is mandatory.** Never schedule two crons at the same minute if they touch overlapping tables. 5-minute spacing minimum.
3. **Prune noise before measuring.** The connection quality metrics were meaningless with 10,655 vector_similar connections inflating counts. After pruning, the numbers told a true story.
4. **Batch, don't cascade.** The M7->M5 propagation flood taught us: when one system's output triggers another system's work, batch the downstream work. Never let a single event cascade through 4+ systems synchronously.
5. **Self-heal weekly, not daily.** cir_self_heal() is expensive (scans all entities). Weekly on Sunday morning is sufficient -- daily monitoring catches urgent issues faster anyway.

---

## Anti-Patterns (Learned)

1. **Do NOT auto-generate connections from embedding similarity alone.** The 10,655 `vector_similar` connections were pure noise -- two companies having similar descriptions does not mean they are related. Only create connections from evidence (interactions, explicit user signals, deal relationships).
2. **Do NOT trust "connection strength > 0.5 = meaningful."** The "85.1% meaningful connections" metric was measuring strength, not evidence quality. 99.9% of connections had single evidence. Strength is a calculated number; evidence count is ground truth.
3. **Do NOT fire all crons at the same time.** 06:00 UTC was a graveyard of connection pool errors. Stagger by 5 minutes.
4. **Do NOT let cascading rescores run unbounded.** A single strategic recalibration was triggering 1,000+ rescore events. Rate limit or batch all downstream propagation.
5. **Do NOT skip embedding dimension verification on new entity types.** WhatsApp came in at 531 dimensions while everything else was 1536. Always verify dimension match when adding a new entity type.
