# M12 — Data Enrichment Machine Context
*Last updated: 2026-03-22. 3 loops completed. Evolving role: identifier + prioritizer -> Datum agent executes.*

---

## Product Vision Context

M12 exists because **data quality is the #1 system bottleneck.** M9 QA scored data quality at 1.5/10 -- 3,800+ companies have less than 500 characters of content (essentially name + one-line description). Every machine downstream is starved: scoring uses thin data, search returns shallow results, Cindy reasons over skeletons, Megamind's strategic analysis has no substance to synthesize.

**What the user cares about:** When Aakash opens a portfolio company page, he expects rich intelligence -- not a name and a one-liner. When he searches for companies in a thesis space, he expects meaningful profiles. When similar companies are shown, they should be accurate and substantive. All of this requires M12 to fill the data gap.

**The evolving role:** M12 started by doing enrichment directly (reading local research files, running SQL updates). It is evolving into a **prioritization and queueing machine** -- M12 identifies WHAT needs enrichment and HOW urgent it is, then queues tasks for the Datum agent to execute autonomously. M12 = triage brain. Datum = execution engine.

## Accumulated Decisions

### Loop 1 — Portfolio Research File Ingestion
- **Matched all 252 portfolio research files** to Supabase company records. 252/252 files matched (100%).
- **Ingested 142 unique portfolio research files** into `companies.page_content` and `portfolio.research_content`. Average file size: 3,000-9,600 characters of structured research.
- **Key insight:** Local research files are the richest data source available. Portfolio companies went from skeletal to having deep research content immediately.
- **Decision:** Portfolio research files are the low-hanging fruit. Non-portfolio companies need web enrichment -- a fundamentally harder problem.

### Loop 2 — Company Creation + Cross-Enrichment
- **Created 84 companies** from research files that referenced companies not in the database. These were competitors, comparables, and ecosystem players mentioned in portfolio research.
- **Cross-enriched 211 companies** with team + thesis data from entity_connections. Companies that had entity connections to people or thesis threads got their page_content augmented with that context.
- **Type and sells_to fields populated:** 33 companies got `type` field filled (SaaS, Marketplace, etc.), 23 got `sells_to` field filled (B2B, B2C, etc.). Previously these were all null.
- **Decision:** Cross-enrichment from existing entity_connections is a cheap win. Every company connected to a thesis or person gets richer for free.

### Loop 3 — Deep Enrichment + Quality Measurement
- **32 companies deep-enriched:** 21 from portfolio research files (structured extraction), 11 from web research (web scraping + synthesis).
- **Quality metrics after L3:** Portfolio companies average 627 chars (up from ~350). 89 companies above 500 chars (up from 66). rich_content_pct = 59%.
- **Pipeline companies identified:** 123 companies in "In Strike Zone" deal status need type/sells_to from web research. These are active deal flow -- enrichment has direct scoring impact.
- **Decision:** Web enrichment is 10-50x slower than local file ingestion but is the only path for 3,800+ non-portfolio companies. This must become Datum's autonomous job.

## Patterns of Success

1. **Local files first, web second.** Portfolio research files yielded 142 deep enrichments in minutes. Web enrichment yielded 11 in the same time. Always exhaust local data before going to web.
2. **Cross-enrichment from entity_connections is free quality.** If a company is connected to a thesis thread, append the thesis context to page_content. If connected to a person, add that relationship context. 211 companies enriched this way with zero API calls.
3. **Measure enrichment by character count thresholds.** <500 chars = skeletal, 500-2000 = moderate, >2000 = rich. This simple metric tracks progress honestly.
4. **Create missing companies from research references.** Research files mention competitors and comparables that may not exist in the database. Creating them expands the searchable universe. 84 companies created this way.
5. **Prioritize by scoring impact.** "In Strike Zone" pipeline companies affect active deal decisions. Portfolio companies affect daily intelligence. Random companies in the long tail can wait.

---

## Anti-Patterns (Learned)

1. **Do NOT count "609 thin companies" when the real number is 4,501.** The initial FB-3 report only counted thin portfolio companies, not the full companies table. M9 QA found the truth: 98.6% of 4,567 companies had <500 chars. Always measure against the FULL entity set.
2. **Do NOT equate "research file matched" with "company enriched."** Matching 252/252 files means the linking is done. Actual enrichment requires reading, extracting, and persisting the content -- a separate step.
3. **Do NOT try to web-enrich 3,800+ companies in one batch.** Web enrichment requires web search + scrape + synthesis per company. At scale, this is Datum agent's autonomous job with rate limiting and quality checks, not a bulk SQL operation.
4. **Do NOT build intelligence logic in M12 SQL functions.** M12 identifies gaps and queues work. Datum agent reasons about HOW to enrich. The enrichment scorecard function (`datum_company_enrichment_scorecard()`) is a measurement tool, not a reasoning engine.

**The #1 system bottleneck is data quality at 1.5/10.** This is the dominant drag on the overall system score (8.3/10 would be higher if data quality rose even to 5/10).

**Scale of the problem:**
- 4,651 total companies

- rich_content_pct = 59% (of PORTFOLIO companies only; system-wide is ~1.9%)

**Next loop priorities (in order):**
1. **Queue 123 "In Strike Zone" companies for Datum web enrichment.** These are active pipeline deals. Enrichment directly impacts scoring and deal intelligence. Create `agent_tasks` entries with task_type = "company_enrichment", priority = high.

3. **Build enrichment prioritization scoring.** Not all 3,800 thin companies are equally important. Priority should be: (a) pipeline companies in active deal status, (b) companies mentioned in recent interactions, (c) companies in active thesis spaces, (d) long tail.
4. **Monitor Datum agent's enrichment throughput.** Once Datum is processing the queue autonomously on the droplet, M12 shifts to monitoring: how many enriched per day, quality of enrichments, failure patterns.
5. **External similar companies web-search refresh.** FB-26 asked for similar companies from external sources, updated monthly. M12 should queue monthly web-search tasks for Datum to find new comparable companies per portfolio entry.
