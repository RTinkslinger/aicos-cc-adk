## Interaction Ownership
Datum owns interaction creation from ALL data sources. When WhatsApp data arrives:
1. **Datum resolves identities** — multi-strategy resolution (exact → company context → fuzzy → unique firstname → nickname patterns)
2. **Datum creates interaction records** — not Cindy, not a SQL function, not a pipeline script
3. **For group chats:** Datum creates BOTH a group-level interaction record (the conversation as a whole) AND per-person extracted interactions (individual contributions, mentions, obligations detected per participant)
4. **lifecycle.py triggers Datum** when new WhatsApp data lands — Datum agent picks up the trigger and reasons about what interactions to create

Datum's SQL functions for resolution (`datum_resolve_whatsapp_v5`, etc.) are TOOLS the agent calls — this is correct since resolution involves deterministic fuzzy matching (utility computation). But the DECISION of whether a match is correct, whether to create an interaction, and what signals to extract = **agent reasoning, not SQL logic**.

## Anti-Patterns
- ILIKE for thesis/company matching (too noisy, false positives)
- SQL auto_process as reasoning brain (should be TOOL for agent)
- Creating garbage network entries (first-name only, zero context)
- SQL functions that DECIDE match correctness (agent decides, SQL proposes candidates)
- Creating interactions without group-level + per-person split for group chats
