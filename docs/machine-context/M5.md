# M5 Scoring — Machine Context
*Accumulated across 6 loops. Product leadership reads at start, writes at end.*

## Product Vision Context
Scoring determines action priority for the user's triage queue. The user sees actions ranked by score and decides what to act on. Good scoring = user trusts the system. Bad scoring = user loses faith and ignores suggestions. Scoring must correctly reflect the user's priority hierarchy: Portfolio + Network actions dominate, Thesis research is agent-delegable.

User cares about: correct priority ordering (portfolio > network > thesis), scoring that matches their intuition, no inflated scores for irrelevant items, clear explanation of why something is scored high/low. Key user signal: "action priorities wrong" (Iteration 29 feedback).

### Scoring is HYBRID — The ONE Legitimate Exception to "No Intelligence in SQL"
Scoring is the one domain where SQL computation is architecturally correct. Here's why:
- **SQL formula for deterministic execution** — same inputs MUST produce same output. `compute_user_priority_score()` is a pure mathematical function: multiply base score by N multipliers, apply sigmoid soft-cap. This is deterministic math, not inference. SQL is the right tool.
- **Agent designs the formula** — which multipliers exist, what weights they carry, how the sigmoid curves, what the priority hierarchy is (portfolio > network > thesis). The agent decides the MODEL. SQL executes the MODEL.
- **Agent can override individual scores** — when contextual reasoning says the formula got it wrong (e.g., user dismissed a high-scoring action because of timing, not relevance), the agent can write an override. The formula is the baseline; the agent is the exception handler.
- **This exception does NOT extend to:** briefing generation, relationship analysis, intelligence summaries, obligation prioritization narratives, or any other function that requires inference/reasoning. Those belong in agents.

## Patterns of Success
- **Multiplicative model** — each multiplier independently adjusts the base score. Easy to diagnose which factor is driving a score up/down.
- **Regression test suite** — 23 automated checks catch regressions before they reach users. Run after every model change.
- **Score snapshots** — versioned snapshots enable drift detection and historical comparison.
- **Semantic key_question matching** via portfolio_key_questions table with embeddings (40.4% coverage, 386 rows).
- **Auto-refresh trigger** — scores refresh on cron (30min) so data changes propagate without manual intervention.
- **Perfect separation** — Accepted (6.78) vs Dismissed (5.03) gap maintained across refreshes.

## Anti-Patterns (Learned)
- **Score ceiling compression** — hard cap at 10.0 caused 53% of actions to cluster at ceiling. Sigmoid soft-cap fixed this.
- **Filtering only `Proposed` status** — causes false negatives when no Proposed actions exist. Always include Accepted + expired.
- **Queue inversion** — Thesis actions dominating top of queue (55%) when Portfolio should dominate. Fixed via priority weighting (portfolio +3, network +2, thesis -3).
- **Rogue normalize crons** — silently ran and corrupted scores. Killed.
- **Score scale corruption** — values on 0-1 scale instead of 0-10. Fixed via convergence inflation bug hunt.
- **Inflated self-grades** — Machine reported 9.6/10 while user experienced 3/10. Only user triage behavior is ground truth.
- **WhatsApp-derived actions missing scoring_factors** — all factors default to 0.5, strategic_score defaults to 0.1. Pipeline must populate these.

## Architecture Note for Other Machines
M5's SQL scoring functions (`compute_user_priority_score()`, `refresh_active_scores()`, `explain_score()`, regression tests) are **legitimate SQL** because they are deterministic mathematical computation. Do NOT refactor these into agent reasoning — that would be wrong. The agent designs the formula; SQL executes it. This is explicitly different from intelligence-generating SQL (briefings, relationship analysis, etc.) which belongs in agents.
