# M6 IRGI — Machine Context
*Accumulated across 4 loops. Product leadership reads at start, writes at end.*

## Product Vision Context
IRGI (Information Retrieval and Graph Intelligence) powers ALL agent reasoning via search and retrieval. When any agent needs context — company background, person history, thesis evidence, interaction patterns — it queries IRGI functions. Search quality = reasoning quality. Every surface the user sees (search page, related entities, holistic profiles) is powered by IRGI.

User cares about: search that works like they think (natural language, cross-surface, disambiguated), related entities that are actually related, fast responses (<500ms). The user types a name or concept and expects the system to find everything relevant across all data surfaces. Key user signal: agents reasoning over thin/wrong data = garbage output.

### Architecture Clarification: Search SQL is Legitimate Infrastructure
Search infrastructure is **legitimate SQL** — FTS indexes, vector similarity (`<=>` operator), trigram matching, multi-surface search routing. This is deterministic computation (given a query, return ranked candidates). This stays in SQL and is correct architecture.

**What stays in SQL (correct):**
- `hybrid_search()`, `balanced_search()`, `person_holistic_search()`, `company_holistic_search()` — deterministic search and retrieval
- FTS indexes, GIN indexes, IVFFlat indexes — infrastructure
- Score computation within search (TF-IDF, cosine similarity, trigram distance) — deterministic math
- WebFront search page calling SQL search functions directly — correct, because search is retrieval not intelligence generation

**What belongs in agents (not SQL):**
- **Disambiguation** — when a search returns multiple candidates and the system must decide which is the correct match, that decision = agent reasoning. SQL proposes candidates with similarity scores; the agent decides which one is right based on context.
- **Search result interpretation** — "what does this search result mean for Aakash's priorities?" = agent reasoning
- **Cross-result synthesis** — combining results from multiple searches into a coherent narrative = agent reasoning

### Loop 3 (Session 2026-03-22 — Rich Detail)
IRGI 9.7/10 (was 9.3). 38/38 benchmarks PASS (was 37/37). 4 changes shipped.

**1. `interaction_thesis_xref` 58x speedup:**
- Problem: 1006ms execution time (was 73ms in L2 when interactions=23, grew to 334 interactions). CROSS JOIN between 334 interactions x 8 theses = 2,672 rows, each running `regexp_split_to_table` for word matching — O(n^2) text splitting.
- Root cause: Quadratic word-by-word comparison via nested `regexp_split_to_table` inside `EXISTS` for every cross-join row.
- Fix (two-part rewrite): (a) LATERAL JOIN with `ORDER BY i.embedding <=> td.embedding LIMIT 30` per thesis leverages IVFFlat index instead of computing all 2,672 distance pairs. (b) FTS tsvector overlap instead of regexp word splitting: pre-compute `to_tsvector('english', string_agg(kq_text))` per thesis once, then check `interaction_fts @@ plainto_tsquery(kq_text)` using existing GIN index.
- Result: Cold 1006ms to 152ms (6.6x), warm 1006ms to 17ms (58x). Cross-join rows 2,672 to 240 (11x fewer). Index usage: IVFFlat + GIN.

**2. `company_holistic_search` — NEW function:**

- 7 enrichment dimensions: portfolio data (health/ownership/scale), connected people (via entity_connections), thesis relevance, recent interactions (linked_companies + summary mention), WhatsApp groups mentioning company, related actions, cross-surface presence count + intelligence completeness score.
- Benchmark: 183ms warm, 311ms cold. PASS (<500ms threshold).
- Test: "AuraML" returns 5/7 surfaces, 0.90 completeness. "E2B" returns 3/7, 0.37. "cybersecurity" returns RSA Security via trigram, 0.62.

**3. WhatsApp search improvements:**

## Patterns of Success
- **LATERAL JOIN + LIMIT for embedding search** — dramatically faster than CROSS JOIN + filter. Leverages IVFFlat indexes.
- **FTS tsvector overlap** — pre-compute tsvectors once per group, then check with `@@`. Uses existing GIN indexes. Far faster than regexp word splitting.
- **Holistic search pattern** — 4 match methods (exact, trigram, FTS, metadata) + 7 enrichment dimensions. Produces rich, multi-surface results.
- **Per-entity-type quotas in balanced_search** — prevents one surface from drowning others. Each entity type gets guaranteed slots.
- **IVFFlat list count = sqrt(row_count)** — 25 lists for 713 rows is optimal. 10 lists was suboptimal.
- **Benchmark suite as regression gate** — 38 tests run after every change. Catches performance regressions immediately.
- **Source-specific enrichment** — WhatsApp, Granola, portfolio each get custom `WHEN` branches in agent_search_context rather than generic handling.

## Anti-Patterns (Learned)
- **CROSS JOIN on growing tables** — interaction_thesis_xref CROSS JOIN grew quadratically (23 interactions = fast, 334 interactions = 1006ms). Always use LATERAL JOIN + LIMIT.
- **regexp_split_to_table for word matching** — O(n^2) text splitting per row. Use FTS tsvector overlap instead.
- **IVFFlat with too few lists** — 10 lists for 637 rows = poor recall. Rebuild when row count grows significantly.
- **NULL enrichment columns** — WhatsApp results returned NULLs for all enrichment because no `WHEN source_table = 'whatsapp'` branch existed. Every new surface needs explicit handling.
- **Flat relevance scores** — all actions at 0.90 makes IRGI useless. Score spread (0.40-0.89) is essential.
- **`relevance_score` holding M5 scores** — IRGI and scoring are independent systems. Each writes to its own column.
- **Threshold set too tight on find_similar_network** — caused benchmark FAIL on legitimate results. Relaxed threshold fixed it.
- **SQL disambiguation** — writing SQL that decides which of multiple search candidates is "correct" based on context. SQL returns ranked candidates; agents decide correctness.
