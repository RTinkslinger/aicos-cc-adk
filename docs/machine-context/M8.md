*Accumulated across loops. Product leadership reads at start, writes at end.*

## Product Vision Context
Cindy is Aakash's intelligent EA -- she tells stories about relationships, not dumps raw data. She is a **persistent Claude Agent SDK agent on the droplet** that REASONS about interactions, detects obligations, extracts action items, surfaces thesis signals, and tracks conversation history with Aakash per person/action. She observes four communication surfaces (Email via AgentMail, WhatsApp, Granola meetings, Calendar).

**Core architecture principle:** Cindy WRITES intelligence outputs to DB tables. WebFront READS those tables via simple SELECTs. No SQL functions generating intelligence. The agent reasons; SQL fetches data.

User cares about: knowing WHO needs attention and WHY (not just "overdue 11 days" but "you offered to connect Ayush with Schneider and haven't followed up"), contextual action options per obligation (not fixed dismiss/reschedule/followup buttons), conversation memory between Cindy and user per action, and portfolio founders being correctly identified.

Cindy's output feeds: M1 WebFront (reads agent-written output tables for /comms page, obligation cards, interaction timeline, relationship momentum, deal velocity), M7 Megamind (obligation blended_priority at 60% Cindy weight, deal signals via cindy_signal), and Datum (unresolved identities, resolution gaps).

## Architecture Overhaul (2026-03-22)

### Current State (WRONG)
46 complex SQL functions ARE the intelligence layer. WebFront calls functions like `cindy_daily_briefing_v3()`, `cindy_relationship_momentum()`, `cindy_deal_velocity()`, `cindy_companies_needing_attention()`, `cindy_relationship_intelligence()` directly. These functions contain scoring logic, priority calculations, intelligence generation, and strategic reasoning in SQL. The Cindy agent is not in the loop -- SQL does the thinking.

### Target State (CORRECT)
Cindy agent (persistent ClaudeSDKClient on droplet) reasons about interactions, obligations, relationships, and deal signals. She writes intelligence outputs to DB tables (briefings, momentum scores, deal signals, attention rankings). WebFront does simple `SELECT * FROM cindy_outputs WHERE type = 'daily_briefing' ORDER BY created_at DESC LIMIT 1`. No SQL generating intelligence. Ever.

### Function Categorization (~46 functions)
- **~25 simple queries (DELETE):** Agent writes its own SQL or these become trivial views. Examples: `cindy_daily_briefing_v3`, `cindy_relationship_momentum`, `cindy_deal_velocity`, `cindy_companies_needing_attention`, `cindy_relationship_intelligence`. The agent reasons about this data and writes structured output.
- **~8 utilities (KEEP as agent tools):** Search, fuzzy match, embedding similarity, phone number lookup, network resolution. Pure data access with no intelligence logic. These are tools the agent calls to gather information before reasoning.
- **~10 complex procedures (DELETE):** `cindy_obligation_auto_generator_v2`, `cindy_recalculate_obligation_priorities`, `cindy_obligation_portfolio_enricher`, `cindy_deal_obligation_generator`. These encode reasoning in SQL -- the agent does this work instead.
- **~3 write operations (DELETE):** `cindy_bridge_whatsapp_to_interactions`, conversation log writes. The agent decides what to write and executes writes directly.

### Lifecycle Trigger
```
WhatsApp upsert (new messages arrive)
  → Datum resolves participants (people matching, confidence >= 0.80)
  → interactions table updated
  → Cindy agent triggered (heartbeat or event)
  → Cindy reasons about new interactions (obligations, signals, momentum, attention)
  → Cindy writes outputs to DB tables
  → WebFront reads output tables on page load (simple SELECT)
```

### Group Chats
Datum creates BOTH:
1. A per-group record in interactions (the group conversation itself)
2. Per-person extracted interactions (what each resolved person said/did in the group)

This gives Cindy person-level signal from group conversations, not just group-level noise.

### Schema Knowledge (Agent Skills)

- Output table schemas she writes to
- Anti-patterns (never put scoring logic in SQL)

## Patterns of Success
- **Multi-strategy resolution.** Cascading strategies (exact, company context, fuzzy, unique firstname, nickname, phone number extraction) achieved 99.7% resolution. Each strategy adds incremental coverage.
- **Bridge WhatsApp to interactions for immediate value.** WhatsApp conversations sitting in `whatsapp_conversations` table were invisible to Cindy. Bridging them to `interactions` via `cindy_bridge_whatsapp_to_interactions()` gave Cindy 19x more data to reason over (23 to 442).
- **Factor-based obligation priorities.** 5-factor scoring (urgency, relationship_depth, portfolio_weight, deal_timing, fund_impact) produces real differentiation. Range 0.245-0.773 vs all-1.0 before.
- **Portfolio enrichment for obligations.** Cross-referencing obligations against portfolio DB catches misclassifications (Surabhi as "Fund Manager" when she is a portfolio founder).
- **Obligation auto-generator with 3 modes** (fetch/process/mark_processed) allows the persistent agent on the droplet to process in batches without re-processing already-seen interactions.

## Anti-Patterns
- **SQL functions generating intelligence = WRONG.** Functions like `cindy_daily_briefing_v3`, `cindy_relationship_momentum`, `cindy_deal_velocity`, `cindy_companies_needing_attention`, `cindy_relationship_intelligence` encode reasoning in SQL. The agent writes briefings, momentum scores, and intelligence outputs -- not SQL procedures. This was the dominant anti-pattern with 46 functions doing what the agent should do.
- **Role from interaction context instead of canonical.** `cindy_deal_obligation_generator()` was picking role_title from interaction context (e.g., "Fund Manager, MSC") instead of `network.role_title` ("Co-Founder CEO"). Always use the canonical source.
- **Portfolio companies classified as deals.** Quivly (pipeline_status=Portfolio) was classified as "active deal negotiations." Must cross-reference portfolio DB before setting obligation category.
- **Deal momentum showing portfolio companies.** `cindy_daily_briefing_v3` deal_signals CTE lacked a NOT EXISTS join to filter portfolio companies. Fixed in M9 L4.
- **Top actions showing cancelled obligations.** `cindy_daily_briefing_v3` active_obligations CTE did not filter cancelled status. Fixed in M9 L4.
- **Procedural CLAUDE.md kills agent reasoning.** 1,004 lines of step-by-step scripts (4 pipeline procedures, 6-tier algorithm, scoring formula) turned the agent into a script runner. Refactored to 224 lines of objectives. The HOW belongs in skill files.
- **All priorities at 1.0 = no signal.** When every obligation has blended_priority=1.0, the user cannot differentiate. Priorities must have meaningful range.
- **WebFront calling SQL functions directly.** WebFront should never call intelligence-generating SQL. It reads agent-written output tables. Simple SELECTs only.
