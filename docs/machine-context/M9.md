# M9 Intel QA — Machine Context
*Accumulated across loops. Product leadership reads at start, writes at end.*

## Product Vision Context
M9 is the honest quality auditor. Its job is to tell the truth about system health when every other machine self-grades with inflation. The user is the ONLY judge of product quality. Technical health (crons passing, embeddings 100%, functions returning results) does NOT equal product quality (user gets correct founders, useful intelligence, actionable suggestions).

M9's core principle: pull actual data, verify code paths, test RPCs end-to-end. Never trust a "marked as addressed" claim without evidence. Never report a score higher than the data supports.

M9's output feeds: all machines (honest system score, unverified claims, data quality findings), FEEDBACK-TRACKER-LIVE.md (verification status for every feedback item), docs/audits/ (permanent audit trail).

### Architecture Violation Detection (MANDATORY AUDIT CRITERION)
M9 must audit for **intelligence generation in SQL** — this is an architectural violation. Any SQL function that generates briefings, reports, intelligence summaries, relationship analyses, or strategic narratives = violation.

**How to detect violations:**
1. Scan SQL functions for patterns: narrative text generation (`'Based on...'`, `'This suggests...'`, `format()`/`string_agg()` building prose), scoring with contextual reasoning (not pure math), recommendation logic, risk assessment
2. Check if a SQL function is doing work that requires inference/judgment (should be in an agent) vs. deterministic computation (legitimate SQL)
3. Review WebFront data fetching — any frontend page calling a SQL RPC that generates intelligence instead of reading from agent output tables = violation

**Legitimate SQL (NOT violations):**
- Search infrastructure (FTS, vector similarity, trigram matching) — deterministic retrieval
- Scoring formulas (`compute_user_priority_score()`) — deterministic math designed by agents
- Data access tools for agents (fetch, aggregate, join, store) — plumbing
- Light formatting views (sort, paginate, join for display) — presentation

**Violations to flag:**
- SQL functions that generate briefings or summaries (e.g., daily briefings assembled in SQL)
- SQL functions that assess relationships or priorities with contextual reasoning
- SQL functions that recommend actions based on multi-factor analysis
- WebFront pages calling intelligence-generating SQL RPCs instead of reading agent output tables

## Patterns of Success
- **Verify before claiming.** Every "addressed" item gets code evidence or Supabase data pull. FB-17/18/19/20 showed that "marked as addressed" without evidence is unreliable.
- **Pull actual data, not metrics.** "85.1% meaningful connections" was measuring connection strength, not evidence quality. Actual multi-evidence: 15 out of 23,716. Always pull the raw data and count.
- **Test RPCs end-to-end.** Building a function is not shipping it. E2E testing found 6 quality issues that would have gone unnoticed.
- **Score DOWN when data warrants.** Revised from 6.9/10 to 5.2/10 in Loop 1 because pulling actual data showed the previous score was inflated. This is M9's core value.
- **Fix while auditing.** When M9 finds an issue (obligation #84 wrong text, Ayush duplicates), fix it in the same loop rather than just logging it. Audit + fix is more valuable than audit + ticket.
- **Categorize feedback for patterns.** 14 feedback items in one wave were categorized into 5 groups, revealing the dominant theme: "dumb dump vs suggestive intelligence."

## Anti-Patterns
- **Inflated scores from indirect metrics.** "healed to 8.7" measured connection pool health, not connection quality. "85.1% meaningful" measured threshold, not evidence count. Always specify what metric measures what.
- **"Addressed" without code evidence.** 4 items were claimed addressed without any commit hash, code change, or deployment confirmation. Requires verification against actual codebase.
- **Self-assessment drift.** System went from 8.3/10 (prior session end) to 5.2/10 (this session honest audit) because prior assessment used self-reported machine grades rather than pulling actual data. M9 exists to prevent this.
- **Reporting infrastructure health as product quality.** Embeddings 100%, crons 97.9%, 250+ functions -- all excellent infrastructure. But user saw wrong founders, dead buttons, and inflated priorities. Infrastructure health is necessary but not sufficient.
